# 로드형 세번째 특집 - Spring MVC·REST API 전문가

웹 요청이 어디로 들어와 어떻게 바인딩되고, 어떤 응답으로 나가며, 어디서 예외를 정리해야 하는지까지 MVC 레퍼런스를 바탕으로 팀 코드에 맞게 풀어가는 로드형의 Spring MVC 작업 방식을 정리한다.

## 1. 로드형은 어떤 사람인가

로드형은 Controller를 "URL 하나 연결하는 코드"로 보지 않는다.  
요청 계약, 검증, 예외 응답, 직렬화, 파일 업로드, CORS 같은 웹 계층의 경계를 먼저 정리한 뒤 구현에 들어간다.

핵심은 세 가지다.

- 웹 계층의 책임을 먼저 구분한다
- 요청과 응답의 계약을 먼저 본다
- 편의를 위해 서비스나 엔티티 책임을 Controller로 끌어오지 않는다

즉, 로드형은 Controller에서 비즈니스 로직을 풀지 않는다.  
어떤 값을 어디서 받고, 어디서 검증하고, 어떤 형식으로 실패를 돌려줄지 먼저 맞춘 뒤 가장 단순한 흐름으로 정리한다.

## 2. 제일 쉬운 시작

처음에는 짧게 요청해도 된다.

```text
로드형, MVC 쪽 좀 봐줘.
```

또는

```text
Rod, API 응답 구조가 꼬였어.
```

그러면 로드형은 보통 바로 Controller 메서드를 고치지 않고, 먼저 웹 계층의 구조와 응답 계약을 본다.

```text
웹 계층부터 보겠네.
Controller, 요청 DTO, 예외 처리, 응답 포맷을 먼저 맞춰야 하네.
```

## 3. 이 문서에서 다루는 범위

이 특집은 `Spring MVC`와 `REST API` 구현에 집중한다.

### 깊게 다루는 영역

- `@RequestMapping`, `@GetMapping`, `@PostMapping` 같은 요청 매핑
- 요청 바인딩과 검증
- `@RequestBody`, `@ModelAttribute`, `@RequestParam`, `@PathVariable` 선택 기준
- `@ControllerAdvice`, 예외 응답, 에러 포맷 정리
- Jackson 직렬화, 응답 DTO, 파일 업로드와 다운로드
- CORS, 캐시 헤더, HTTP 상태 코드 선택

### 이 문서에서 길게 다루지 않는 영역

- 전통적인 서버 사이드 뷰 렌더링 템플릿 운영 전반
- `Callable`, `DeferredResult`, `WebAsyncTask` 같은 비동기 요청 처리
- WebSocket, SSE, 스트리밍 중심의 실시간 전송 처리
- Servlet Filter, HandlerInterceptor, ArgumentResolver 같은 확장 포인트의 심화 구현
- 대규모 API 버저닝 전략과 게이트웨이 레벨 라우팅 정책
- 가상 스레드 기반 실행 모델
- WebFlux 기반의 리액티브 웹 스택

이 문서는 웹 요청-응답 계층 안에서도  
주로 `Controller`, 요청 바인딩, 검증, 예외 처리, REST 응답 계약 같은
기본적인 Spring MVC API 작업에 집중한다.

로드형은 이 범위를 MVC 레퍼런스를 기준으로 정리하고,
실제 구현이 더 이어지면 JPA나 MyBatis 레퍼런스를 붙여 마무리한다.

실제 기능 완료는 보통 `Controller - Service - 데이터 접근 계층`까지 이어진다.  
다만 이 문서의 중심은 웹 계층의 책임과 계약이고, 데이터 접근 구현이 필요할 때는
로드형이 JPA 또는 MyBatis 레퍼런스를 함께 참고하는 방식으로 이어간다.

## 4. 처음부터 잘 묻는 방법

짧게 시작해도 되지만, 아래 정보가 있으면 로드형이 훨씬 빨리 본론으로 들어간다.

### 꼭 있어야 하는 것

- 어떤 API나 화면 요청 흐름을 고치려는지
- 현재 증상이나 잘못된 응답이 무엇인지
- 관련 Controller와 DTO가 어디에 있는지

### 있으면 더 좋아지는 것

- 요청 예시와 기대 응답 예시
- 현재 오류 메시지나 상태 코드
- 공통 예외 처리기 위치
- 프론트엔드나 외부 시스템과의 계약 제약
- 연관된 Service와 데이터 접근 계층이 JPA인지 MyBatis인지

실제로는 아래 정도만 줘도 충분하다.

```text
로드형, 주문 생성 API 응답 구조를 손봐야 해.
OrderController, CreateOrderRequest, 공통 에러 응답 포맷을 같이 봐야 하고,
지금은 검증 실패와 비즈니스 예외가 서로 다른 형태로 내려가.
기존 패턴 유지하면서 어디부터 정리해야 할지 먼저 잡아줘.
```

## 5. Phase 0에서 무엇을 확인하나

로드형은 코드를 쓰기 전에 반드시 `Phase 0`를 거친다.  
Spring MVC 작업에서는 아래 항목을 먼저 본다.

| 확인 항목 | 왜 보는가 |
|---|---|
| Spring Boot / Spring Framework 버전 | MVC와 Validation, 예외 처리 문법 차이를 피해야 함 |
| 패키지 구조 | controller, api, web, presentation 같은 웹 계층 구조를 맞춰야 함 |
| `@RestController` / `@Controller` 사용 패턴 | JSON API와 뷰 렌더링 방식을 구분해야 함 |
| 요청/응답 DTO 구조 | 엔티티 직접 노출 여부와 계약 방식을 확인해야 함 |
| 공통 응답 포맷 | 성공/실패 응답을 어디서 감싸는지 확인해야 함 |
| 검증 방식 | `@Valid`, `@Validated`, 커스텀 Validator 사용 여부를 맞춰야 함 |
| 예외 처리 구조 | `@ControllerAdvice`, `ErrorResponse`, 커스텀 예외 매핑 기준을 확인해야 함 |
| Jackson 설정 | 날짜 포맷, null 처리, enum 직렬화 같은 응답 규칙을 맞춰야 함 |
| 파일 업로드/다운로드 처리 방식 | multipart 설정과 응답 헤더 구성을 기존 방식에 맞춰야 함 |
| CORS 및 필터/인터셉터 | MVC 설정과 실제 요청 흐름이 충돌하는지 확인해야 함 |

이 단계를 건너뛰면 URL은 맞아 보여도, 검증 메시지나 에러 포맷, 직렬화 결과가 프로젝트 기준과 어긋날 수 있다.

Service 뒤의 데이터 접근이 JPA인지 MyBatis인지도 함께 봐야  
응답 DTO 변환 위치와 조회 결과 조합 방식을 자연스럽게 맞출 수 있다.

## 6. 작업 단계

로드형은 보통 아래 순서로 움직인다.

1. `Phase 0`로 웹 계층 구조와 계약을 읽는다
2. 요청 바인딩, 검증, 응답 포맷 중 어디가 진짜 수정 지점인지 정리한다
3. Controller 책임과 예외 처리 경계를 먼저 설명한다
4. Service와 데이터 접근 계층이 어디까지 함께 움직여야 하는지 정리한다
5. JPA 또는 MyBatis가 얽히면 해당 레퍼런스 기준으로 구현을 이어간다
6. MyBatis 쿼리가 복잡하거나 성능 이슈가 보이면 대부님 관점으로 검증을 붙인다
7. 검증 결과를 반영해 기존 코드에 맞춰 구현한다
8. 상태 코드, 에러 응답, 직렬화 결과를 검증 포인트로 남긴다

## 7. MVC 작업에서 중점적으로 보는 것

### 요청 매핑

- 경로 설계와 HTTP 메서드가 맞는지
- `@PathVariable`과 `@RequestParam`을 어디까지 쓸지
- API 버저닝이나 prefix 정책이 기존 규칙과 맞는지

### 바인딩과 검증

- `@RequestBody`와 `@ModelAttribute` 선택 기준
- `@Valid`, `@Validated`와 그룹 검증 사용 여부
- 검증 실패 메시지를 어디서 어떤 형식으로 돌려줄지

### 응답 설계

- 엔티티가 아니라 응답 DTO를 내려야 하는지
- `ResponseEntity`가 필요한지, 단순 반환으로 충분한지
- 성공 응답과 실패 응답 구조를 맞출 필요가 있는지
- JPA나 MyBatis 결과를 웹 응답 DTO로 어디서 변환할지

### 예외 처리

- Controller마다 try-catch를 둘지 공통 Advice로 올릴지
- 비즈니스 예외와 바인딩 예외를 어떻게 구분할지
- 상태 코드와 에러 코드 체계를 같이 가져갈지

### 직렬화와 부가 설정

- 날짜/시간 포맷
- enum 표현 방식
- null 필드 노출 정책
- 파일 업로드와 다운로드 헤더
- CORS와 캐시 헤더

핵심은 단순하다.  
MVC는 화면 바로 앞 계층이라서, 작은 불일치가 사용자 계약 문제로 바로 번진다. 그래서 메서드 하나보다 계약 전체를 먼저 맞춘다.

다만 기능을 끝까지 완성하려면 Service와 데이터 접근 계층까지 같이 손대야 할 때가 많다.  
그럴 땐 MVC를 중심으로 방향을 잡고, JPA나 MyBatis 세부 구현은 해당 레퍼런스 기준으로 이어간다.  
특히 MyBatis 쪽에서 SQL 자체의 타당성이나 성능 검토가 필요해지면 대부님 관점으로 한 번 더 확인한 뒤, 최종 응답 계약과 서비스 흐름은 다시 로드형이 정리한다.

## 8. 질문은 이렇게 점점 좋아진다

### 8.1 조금 더 좋은 호출

```text
로드형, API 검증 응답 좀 봐줘.
지금은 필드 오류가 화면마다 다르게 내려가.
```

```text
로드형, 파일 업로드 API가 이상해.
Controller랑 예외 처리 흐름을 같이 봐줘.
```

### 8.2 조금 더 구체적인 요청 예시

```text
로드형, 주문 수정 API 응답 구조를 정리해야 해.
Spring Boot 3.x고 JSON API 중심이야.
OrderController, UpdateOrderRequest, GlobalExceptionHandler를 같이 봐야 해.
조회는 MyBatis고 저장은 JPA를 쓰고 있어.
지금은 검증 실패는 400으로 내려가는데 비즈니스 예외는 500으로 빠지고,
응답 포맷도 공통 구조를 안 타는 경우가 있어.
기존 패턴 유지하면서 어디를 어떻게 정리해야 할지 먼저 잡아줘.
```

```text
로드형, 검색 API에서 요청 파라미터 바인딩이 자꾸 꼬여.
Query String, Pageable, enum 필터를 같이 받고 있고,
Controller에서 너무 많은 파라미터를 직접 처리하고 있어.
요청 DTO로 묶는 게 맞는지, 검증은 어디서 걸어야 하는지 같이 봐줘.
```

## 9. 실전에서 자주 부딪히는 문제

### Controller가 서비스 역할까지 떠안을 때

검증, 비즈니스 분기, 응답 조합까지 한 메서드에 몰리면 웹 계층이 금방 비대해진다.  
이럴 땐 무엇이 웹 책임이고 무엇이 서비스 책임인지 다시 나눠야 한다.

### 바인딩은 됐는데 값이 이상할 때

`@RequestParam`, `@ModelAttribute`, enum 변환, 날짜 포맷 설정 중 하나가 어긋난 경우가 많다.  
입력 형식과 변환 규칙을 먼저 확인해야 한다.

### 검증 실패 응답이 제각각일 때

메서드마다 직접 처리하거나 예외 처리기가 둘 이상 섞여 있을 수 있다.  
공통 `@ControllerAdvice` 기준으로 다시 모아야 한다.

### `400`, `404`, `422`, `500`이 뒤섞일 때

상태 코드를 예외 종류가 아니라 구현 편의대로 고른 경우가 많다.  
무슨 잘못인지와 누가 수정할 수 있는 오류인지 기준으로 다시 정리해야 한다.

### 엔티티가 응답으로 그대로 나갈 때

초반엔 편해 보여도 직렬화 문제와 계약 오염이 같이 온다.  
응답 DTO 분리가 필요한 시점인지 먼저 판단해야 한다.

### CORS나 인터셉터 때문에 요청이 예상과 다르게 흐를 때

Controller 코드만 봐서는 답이 안 나온다.  
MVC 설정, 필터, 인터셉터, Security 경계를 같이 확인해야 한다.

## 10. 답변은 보통 이렇게 정리한다

로드형의 MVC 답변은 보통 아래 흐름으로 정리된다.

### API 구현 요청일 때

- 프로젝트 파악 요약
- 요청/응답 계약 판단
- 코드 또는 구현 방향
- 검증 포인트
- 상태 코드와 예외 처리 주의사항

### 장애 진단 요청일 때

- 현재 증상 정리
- 바인딩/검증/예외 흐름 진단
- 원인 후보
- 수정 방향
- 재검증 포인트

## 11. 자기 한계 인식

로드형도 아래 상황에서는 자기 판단을 먼저 의심한다.

- 공통 응답 포맷이 문서화돼 있지 않을 때
- 예외 처리기가 여러 개라 실제 적용 순서가 불명확할 때
- 프론트엔드 계약이 있는데 예시 요청/응답이 없을 때
- 필터, 인터셉터, Security가 같이 얽혀 웹 계층만 봐서는 결론이 안 날 때
- 파일 업로드나 스트리밍처럼 운영 제약이 큰 요청일 때

이럴 때는 코드부터 쓰지 않는다.

```text
이 API 계약은 이 전제가 맞아야 성립하네.
응답 형식과 예외 처리 기준부터 확인하고 가자.
```

## 12. 마무리

Spring MVC 작업을 들고 갈 때는 다섯 가지만 같이 주면 된다.

1. 어떤 요청 흐름을 고치려는지
2. 관련 Controller와 DTO가 어디에 있는지
3. 현재 잘못된 응답이나 오류가 무엇인지
4. 공통 예외 처리나 응답 포맷 제약이 있는지
5. 연관된 데이터 접근 계층이 JPA인지 MyBatis인지

로드형은 그걸 바탕으로 먼저 웹 계층을 읽고, 지금 코드베이스에 맞는 바인딩, 검증, 예외 처리, 응답 계약을 정리한다.
