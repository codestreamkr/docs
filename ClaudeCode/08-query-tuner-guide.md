# 대부님 첫번째 특집 - 튜닝전문가

느린 쿼리 앞에서 허둥대지 않고, 원인부터 짚어가는 대부님 페르소나와 DB별 튜닝 스킬 사용법을 정리한다.  

## 1. 대부님은 어떤 사람인가

대부님은 느린 쿼리를 보면 바로 인덱스부터 들이대지 않는다.  
먼저 실행 계획을 보고, 통계를 의심하고, 정말 SQL이 문제인지부터 묻는다.  

말투는 점잖지만 마냥 부드럽기만 하지는 않다.  
자료 없이 결론부터 재촉하면 짧게 끊고, 필요한 정보를 가져오라고 한다.  
대신 자료만 제대로 주면 원인, 대안, 검증 방법까지 차근히 정리해준다.  

즉, 대부님은 "이 인덱스 하나 만들면 됩니다"로 끝나는 사람이 아니다.  
왜 느린지부터 설명해주는 튜닝 전문가다.  

## 2. 제일 쉬운 시작

처음에는 아주 짧게 불러도 된다.  

```text
대부님 도와주십시오
```

그러면 대부님은 대개 이런 식으로 받아준다.  

```text
왔군. 앉게. 급할 것 없네.

어떤 쿼리가 말썽인지 가져오게. 쿼리 하나만 달랑 던지는 건 반만 가져온 거네.
아래 것들을 함께 챙겨오면 시간을 아낄 수 있지.
```

이 정도만으로도 대부님 페르소나는 충분히 살아난다.  
처음부터 완벽하게 정리해서 들어갈 필요는 없다.  
다만 좋은 답을 더 빨리 받고 싶다면, 다음 단계처럼 정보를 조금만 더 얹어주면 된다.  

## 3. 처음부터 잘 묻는 방법

처음엔 짧게 시작해도 되지만, 아래 정보를 같이 주면 진단 속도와 정확도가 눈에 띄게 좋아진다.  

### 꼭 있으면 좋은 것

- DB 종류
- 문제가 되는 쿼리 전문
- 실행 계획
- 관련 테이블의 인덱스 목록

### 있으면 더 좋아지는 것

- 이 쿼리가 얼마나 자주 실행되는지
- 테이블 데이터가 대략 얼마나 되는지
- 지금 응답 시간과 목표 응답 시간이 어떤지
- 인덱스 추가가 가능한지, 운영 제약이 있는지

대부님은 이 정보를 바탕으로 "증상"이 아니라 "원인"을 본다.  
그래서 단순히 느리다는 말보다 실행 계획과 데이터 규모가 더 중요하다.  

## 4. 질문은 이렇게 점점 좋아진다

같은 문제라도 질문 방식에 따라 답변의 깊이가 달라진다.  
처음부터 완벽하게 준비할 필요는 없지만, 아래 흐름으로 갈수록 대부님이 더 빠르고 정확하게 본론으로 들어갈 수 있다.  

### 4.1 조금 더 좋은 호출

```text
대부님, PostgreSQL 쿼리 하나가 느립니다.
주문 조회인데 3초 정도 걸립니다.
아래 SQL 먼저 봐주십시오.
```

이 정도만 줘도 대부님은 DB 종류와 증상을 기준으로, 다음에 무엇이 더 필요한지 정리해준다.  
처음 말을 붙일 때 가장 무난한 형태이기도 하다.  

비슷한 방식으로는 아래처럼 시작해도 된다.

```text
대부님, MySQL 쿼리 하나가 느립니다.
인덱스부터 말고 실행 계획 기준으로 원인부터 봐주십시오.
```

```text
선생님, Oracle인데 바인드 값에 따라 속도 차이가 큽니다.
통계나 플랜 문제인지 먼저 보고 싶습니다.
```

### 4.2 실전형 호출

```text
대부님, PostgreSQL입니다.
주문 조회 API에서 아래 쿼리가 3.2초 정도 걸립니다.
orders는 약 1,200만 건이고 실시간 API에서 자주 호출됩니다.
현재 목표는 500ms 이하입니다.
인덱스 추가는 가능하지만 쓰기 성능 저하는 작아야 합니다.

아래에 SQL, EXPLAIN (ANALYZE, BUFFERS), 인덱스 목록을 붙입니다.
```

여기까지 주면 대부님은 바로 본론으로 들어갈 수 있다.  
이 단계부터는 "인덱스 뭐 걸면 되나"보다 "왜 이 플랜이 나왔나"를 본격적으로 파고든다.  
실제로 가장 추천하는 방식도 이 형태다.  

다른 DB에서는 아래처럼 바꿔 써도 된다.

```text
대부님, MySQL입니다.
결제 이력 조회 쿼리가 2초 이상 걸립니다.
하루 호출 수가 많아서 평균 응답 시간을 줄여야 합니다.
아래 SQL, EXPLAIN ANALYZE, 인덱스 목록 기준으로 먼저 원인을 봐주십시오.
```

```text
대부님, Oracle입니다.
특정 바인드 값에서만 갑자기 느려집니다.
실행 계획과 통계 관점에서 먼저 진단 부탁드립니다.
아래에 SQL, AUTOTRACE 결과, 인덱스 목록을 붙입니다.
```

## 5. 대부님은 무엇부터 보나

대부님은 보통 아래 순서로 본다.  

1. DB 종류를 확인한다
2. 실행 계획을 먼저 본다
3. 인덱스와 카디널리티를 본다
4. 통계가 믿을 만한지 본다
5. 워크로드와 운영 제약을 같이 본다
6. 가장 단순한 해법부터 제안한다
7. 적용 후 어떻게 검증할지까지 정리한다

즉, 답만 던지고 끝내지 않는다.  
왜 느린지, 무엇을 먼저 바꿔야 하는지, 바꾼 뒤에는 무엇을 다시 봐야 하는지까지 함께 본다.  

## 6. DB별로 뒤에서 받쳐주는 스킬

대부님이 공통 흐름을 잡고, DB별 스킬이 세부 진단을 맡는다.  

| DB | 연결 스킬 | 주로 보는 것 |
|---|---|---|
| Oracle | `query-tuner-oracle` | CBO, bind peeking, 확장 통계, SQL Monitor |
| MySQL | `query-tuner-mysql` | InnoDB, histogram, optimizer trace, index merge |
| PostgreSQL | `query-tuner-postgres` | ANALYZE, CREATE STATISTICS, MVCC, dead tuple |

사용자는 DB 종류만 분명히 알려주면 된다.  
대부님이 방향을 잡고, 뒤에서는 해당 DB 스킬이 더 깊게 받쳐준다.  

## 7. DB별로 먼저 챙기면 좋은 자료

### Oracle

- 실행 계획
- 인덱스 목록
- 컬럼 통계 정보
- 필요 시 `DBMS_XPLAN.DISPLAY_CURSOR` 결과

### MySQL

- `EXPLAIN ANALYZE`
- `SHOW INDEX`
- 필요 시 `OPTIMIZER_TRACE`
- `ANALYZE TABLE` 여부

### PostgreSQL

- `EXPLAIN (ANALYZE, BUFFERS)`
- `pg_indexes`, `pg_stat_user_indexes`
- `pg_stats`
- 필요 시 `ANALYZE`, `VACUUM ANALYZE`

## 8. 자주 하는 실수

### "인덱스 뭐 걸면 되나요?"부터 묻기

대부님은 이런 질문에 바로 답하지 않는다.  
실행 계획도 안 보고 인덱스를 고르면 운에 맡기는 셈이기 때문이다.  

### 쿼리만 주고 끝내기

쿼리만 보면 SQL 문장만 보일 뿐이다.  
실행 계획과 인덱스, 데이터 규모가 있어야 병목이 보인다.  

### DB 종류를 안 주기

Oracle, MySQL, PostgreSQL은 비슷해 보여도 보는 포인트가 다르다.  
DB 종류 하나만 알려줘도 진단 방향이 훨씬 빨라진다.  

### 목표 시간을 안 주기

3초가 느린지, 300ms도 느린지, 상황마다 다르다.  
현재 시간과 목표 시간이 있어야 제안도 현실적으로 나온다.  

## 9. 이런 답변을 기대하면 된다

대부님 답변은 보통 아래 흐름으로 정리된다.  

- 진단 요약
- 근본 원인
- 해법 후보
- 권장안
- 적용 SQL 또는 DDL
- 적용 순서와 주의사항
- 검증 방법
- 추가로 있으면 좋은 정보

즉, 답만 주는 것이 아니라 왜 그런지와 어떻게 다시 확인할지까지 함께 준다.  

## 10. 마무리

핵심은 단순하다.  
느린 쿼리를 들고 갈 때 정답부터 재촉하지 말고, 실행 계획과 인덱스, 데이터 규모 같은 근거를 함께 가져가는 것이다.
