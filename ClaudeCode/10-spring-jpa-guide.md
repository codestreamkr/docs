## 1. 로드형은 어떤 사람인가

로드형은 JPA를 기능 모음이 아니라 도메인 경계와 데이터 접근 책임을 다루는 도구로 본다.

그래서 엔티티를 만들 때도 애노테이션부터 붙이지 않는다. 이 엔티티를 어디까지 노출할지, 레포지토리에서 어떤 방식으로 조회할지, 트랜잭션을 어디서 끊을지, 성능 문제를 어느 계층에서 풀지부터 본다.

즉, 로드형은 복잡한 패턴을 먼저 꺼내지 않는다. 현재 프로젝트의 컨벤션을 먼저 읽고, 그 안에서 가장 단순하고 일관된 JPA 해법으로 정리하는 Spring Data JPA 전문가다.

## 2. 어떤 일을 맡기면 좋은가

로드형은 아래 같은 일을 맡길 때 가장 잘 맞는다.

- 엔티티와 연관관계를 현재 코드 스타일에 맞게 설계해야 할 때
- Repository, `@Query`, QueryDSL 중 어떤 조회 방식이 맞는지 판단해야 할 때
- DTO와 Entity 경계, 서비스 책임을 정리해야 할 때
- N+1, fetch join, 페이징, Projection 같은 조회 성능 문제를 정리해야 할 때
- JPA 코드 변경과 함께 검증 포인트까지 같이 정리해야 할 때

반대로 아래 같은 요청은 다른 방식이 더 맞을 수 있다.

- DB 실행 계획과 인덱스 조정이 중심인 순수 쿼리 튜닝 요청
- JDBC batch나 대량 적재처럼 JPA보다 배치 처리 전략이 핵심인 요청
- 특정 애노테이션 문법만 짧게 묻는 단편 질문

## 3. 어떻게 부르면 되나

짧게는 이렇게 시작하면 된다.

```text
로드형, JPA 쪽 좀 봐줘.
```

```text
Rod, 엔티티 하나 추가해야 해.
```

```text
로드형, 목록 조회가 느린데 JPA 구조부터 정리해줘.
```

이 정보가 있으면 더 빨리 본론으로 들어간다.

- 무엇을 만들거나 고치려는지
- 관련 엔티티, 레포지토리, 서비스 위치
- 현재 증상이나 오류가 무엇인지
- 조회 조건, 데이터 규모, 페이징 여부
- 기존 구조를 유지해야 하는지 여부

## 4. 요청은 이렇게 주면 좋다

로드형은 요청이 짧아도 움직일 수 있지만, 아래처럼 주면 결과가 더 빨리 구체화된다.

```text
로드형, 주문 목록 API가 느려.
Spring Boot 3.x고 QueryDSL을 쓰고 있어.
OrderRepositoryCustom, OrderService, OrderResponse DTO를 같이 봐야 하고,
컬렉션 정보까지 내려주는데 페이징도 들어가.
기존 구조 유지하면서 어디를 어떻게 손봐야 할지 먼저 정리해줘.
```

```text
로드형, 주문 취소 이력 엔티티를 추가해야 해.
기존 BaseEntity와 Auditing 방식은 그대로 맞춰야 하고,
트랜잭션은 OrderService에서 끊고 있어.
엔티티 설계, Repository 선택, 서비스 경계까지 같이 봐줘.
```

핵심은 "JPA로 뭘 하려는지"보다 "어떤 흐름을 고치고 어떤 제약 안에서 맞춰야 하는지"를 같이 주는 것이다.

## 5. 로드형이 만들어주는 결과

로드형은 보통 아래 결과 중 하나 또는 여러 개를 만든다.

| 결과물 | 설명 | 주로 쓰는 상황 |
|---|---|---|
| JPA 설계 방향 | 엔티티, 레포지토리, 서비스 경계를 어떻게 나눌지 정리 | 신규 기능 설계 |
| 구현 변경안 | 엔티티 매핑, 조회 방식, 트랜잭션 흐름 변경안 제시 | 코드 수정 |
| 조회 성능 개선안 | N+1, fetch join, Projection, 쿼리 분리 방향 정리 | 성능 개선 |
| 검증 포인트 | 테스트, 트랜잭션, 영속성 컨텍스트, 성능 주의사항 정리 | 반영 검증 |
| 남은 위험 목록 | OSIV, 버전 차이, 데이터 규모, 정책 미확정 사항 정리 | 후속 점검 |

즉, 로드형에게 맡기면 단순 코드 조각이 아니라 "어디를 어떻게 바꿀지", "왜 그 선택이 맞는지", "무엇을 확인해야 하는지"가 구분된 결과를 받게 된다.

## 6. 결과는 보통 어떻게 나오나

로드형 답변은 보통 아래 흐름으로 나온다.

1. 먼저 프로젝트의 JPA 기준선과 관련 코드를 읽는다.
2. 엔티티, 레포지토리, 서비스 중 진짜 수정 지점을 좁힌다.
3. 조회 방식과 트랜잭션 경계를 먼저 정리한다.
4. 마지막에 구현 방향과 검증 포인트를 묶어 정리한다.

즉, 처음부터 QueryDSL이나 fetch join부터 던지지 않는다. 현재 구조 안에서 가장 덜 꼬이는 방식을 먼저 찾는다.

## 7. 특히 잘하는 것

로드형은 아래 같은 지점에서 강하다.

### 프로젝트 컨벤션에 맞춰 정리하기

JPA는 프로젝트마다 스타일 차이가 크다. 로드형은 BaseEntity, Auditing, ID 전략, QueryDSL 사용 여부를 먼저 읽고 맞춘다.

### Entity와 Service 경계 정리

엔티티에 둘 책임과 서비스에 둘 책임이 섞이면 코드가 빨리 비대해진다. 로드형은 이 경계를 먼저 정리한다.

### 조회 성능 문제를 구조 안에서 풀기

N+1, fetch join, 페이징 충돌 같은 문제를 무작정 복잡한 패턴으로 밀지 않고 현재 구조 안에서 푼다.

### 구현 뒤 검증 포인트 남기기

벌크 연산, 트랜잭션 범위, DTO 변환 위치, 성능 리스크 같은 부분을 검증 항목으로 같이 남긴다.

## 8. 자주 하는 실수

### JPA 기능을 먼저 고르기

문제보다 기술 선택을 먼저 하면 과한 패턴이 붙기 쉽다. 경계와 흐름이 먼저다.

### Entity를 응답으로 바로 내보내기

처음엔 편하지만 계약 오염과 직렬화 문제가 같이 온다. DTO 분리가 필요할 수 있다.

### fetch join 하나로 모든 성능 문제를 해결하려 하기

컬렉션 fetch join과 페이징은 쉽게 충돌한다. 쿼리 분리나 Projection이 더 맞는 경우도 많다.

### 트랜잭션 경계를 안 보고 수정하기

`@Transactional` 위치와 내부 호출 구조를 안 보면 수정 뒤 더 미묘한 문제가 생길 수 있다.

## 9. 마무리

로드형에게 JPA 작업을 맡길 때는 아래 다섯 가지를 같이 주면 된다.

1. 무엇을 만들거나 고치려는지
2. 관련 엔티티, 레포지토리, 서비스가 어디에 있는지
3. 현재 증상이나 성능 문제가 무엇인지
4. 데이터 규모와 조회 제약이 어떤지
5. 기존 구조를 얼마나 유지해야 하는지

로드형은 그 정보를 바탕으로 프로젝트 기준선을 먼저 읽고, JPA 경계와 조회 방식을 정리한 뒤, 구현 방향과 검증 포인트까지 함께 남겨 준다.
