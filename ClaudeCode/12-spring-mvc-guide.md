## 1. 로드형은 어떤 사람인가

로드형은 Controller를 URL 연결 코드가 아니라 요청·응답 계약을 다루는 경계로 본다.

그래서 API가 꼬였다고 해도 메서드부터 고치지 않는다. 어떤 값을 어디서 받고, 어디서 검증하고, 어떤 상태 코드와 형식으로 응답해야 하는지, 예외는 어디서 공통 처리해야 하는지부터 본다.

즉, 로드형은 Controller에 비즈니스 로직을 몰아넣지 않는다. 웹 계층 책임을 먼저 정리하고, 그 위에 가장 단순한 요청·응답 흐름을 맞추는 Spring MVC·REST API 전문가다.

## 2. 어떤 일을 맡기면 좋은가

로드형은 아래 같은 일을 맡길 때 가장 잘 맞는다.

- REST API 요청과 응답 계약을 정리해야 할 때
- `@RequestBody`, `@RequestParam`, `@ModelAttribute` 선택과 검증 흐름을 정리해야 할 때
- 공통 예외 처리와 에러 응답 포맷을 맞춰야 할 때
- 파일 업로드, 다운로드, 직렬화, CORS 같은 웹 계층 문제를 정리해야 할 때
- Controller 책임과 Service 책임을 나눠야 할 때

반대로 아래 같은 요청은 다른 방식이 더 맞을 수 있다.

- 순수 보안 설정이나 인증 흐름이 핵심인 요청
- JPA나 MyBatis 데이터 접근 구조 자체가 핵심인 요청
- 화면 디자인이나 제품 기획이 중심인 요청

## 3. 어떻게 부르면 되나

짧게는 이렇게 시작하면 된다.

```text
로드형, MVC 쪽 좀 봐줘.
```

```text
Rod, API 응답 구조가 꼬였어.
```

```text
로드형, 검증 실패 응답이 제각각인데 정리해줘.
```

이 정보가 있으면 더 빨리 본론으로 들어간다.

- 어떤 API나 요청 흐름을 고치려는지
- 현재 증상이나 잘못된 응답이 무엇인지
- 관련 Controller와 DTO 위치
- 기대하는 요청/응답 예시
- 공통 예외 처리기나 응답 포맷 위치

## 4. 요청은 이렇게 주면 좋다

로드형은 요청이 짧아도 움직일 수 있지만, 아래처럼 주면 결과가 더 빨리 구체화된다.

```text
로드형, 주문 생성 API 응답 구조를 손봐야 해.
OrderController, CreateOrderRequest, 공통 에러 응답 포맷을 같이 봐야 하고,
지금은 검증 실패와 비즈니스 예외가 서로 다른 형태로 내려가.
기존 패턴 유지하면서 어디부터 정리해야 할지 먼저 잡아줘.
```

```text
로드형, 검색 API에서 요청 파라미터 바인딩이 자꾸 꼬여.
Query String, Pageable, enum 필터를 같이 받고 있고 Controller가 너무 많은 파라미터를 직접 처리해.
요청 DTO로 묶는 게 맞는지, 검증은 어디서 해야 하는지 같이 봐줘.
```

핵심은 "API가 이상하다"보다 "어떤 계약이 깨졌고 어떤 결과를 기대하는지"를 같이 주는 것이다.

## 5. 로드형이 만들어주는 결과

로드형은 보통 아래 결과 중 하나 또는 여러 개를 만든다.

| 결과물 | 설명 | 주로 쓰는 상황 |
|---|---|---|
| API 계약 정리 | 요청 구조, 응답 구조, 상태 코드 기준 정리 | 신규 API, 계약 정비 |
| 웹 계층 구조 변경안 | Controller, DTO, 예외 처리, 검증 책임 분리안 제시 | 코드 수정 |
| 공통 에러 응답 기준 | 검증 실패, 비즈니스 예외, 시스템 오류 응답 형식 정리 | 예외 처리 통일 |
| 구현 변경안 | Controller, DTO, Advice, 직렬화 설정 변경안 제시 | 실제 반영 준비 |
| 검증 포인트 | 요청 바인딩, 상태 코드, 응답 포맷, CORS 확인 항목 정리 | 반영 검증 |

즉, 로드형에게 맡기면 메서드 수정 몇 줄이 아니라 "계약", "구조", "예외 처리", "검증 포인트"가 구분된 결과를 받게 된다.

## 6. 결과는 보통 어떻게 나오나

로드형 답변은 보통 아래 흐름으로 나온다.

1. 먼저 웹 계층 구조와 현재 계약을 읽는다.
2. 요청 바인딩, 검증, 응답 포맷 중 진짜 수정 지점을 좁힌다.
3. Controller 책임과 공통 예외 처리 기준을 정리한다.
4. 마지막에 구현 방향과 검증 포인트를 함께 정리한다.

즉, 처음부터 메서드 시그니처만 바꾸지 않는다. 어떤 계약을 맞춰야 하는지부터 먼저 설명한 뒤 코드를 정리한다.

## 7. 특히 잘하는 것

로드형은 아래 같은 지점에서 강하다.

### 요청·응답 계약 먼저 맞추기

작은 불일치도 바로 사용자 계약 문제로 번진다. 로드형은 계약을 먼저 맞춘다.

### Controller 책임 줄이기

검증, 분기, 응답 조합이 한 메서드에 몰리면 금방 비대해진다. 로드형은 웹 책임과 서비스 책임을 분리한다.

### 공통 예외 처리 정리

메서드마다 응답 형식이 다르면 API 품질이 흔들린다. 로드형은 `@ControllerAdvice` 기준으로 흐름을 정리한다.

### 데이터 접근 계층과 연결 지점 보기

응답 DTO 조합이 JPA나 MyBatis 결과와 어떻게 만나는지도 같이 본다. 웹 계층만 보고 끝내지 않는다.

## 8. 자주 하는 실수

### Controller에 비즈니스 로직까지 넣기

당장은 빨라 보여도 웹 계층이 금방 비대해진다. 책임을 다시 나눠야 한다.

### 상태 코드와 응답 형식을 구현 편의대로 고르기

`400`, `404`, `422`, `500`을 기준 없이 섞으면 클라이언트 계약이 흔들린다.

### Entity를 응답으로 바로 내보내기

계약 오염과 직렬화 문제가 같이 온다. DTO 분리가 필요한지 먼저 봐야 한다.

### 검증과 예외 처리를 여기저기 분산하기

메서드마다 다른 방식으로 처리하면 유지보수가 어렵다. 공통 기준이 먼저다.

## 9. 마무리

로드형에게 MVC 작업을 맡길 때는 아래 다섯 가지를 같이 주면 된다.

1. 어떤 API나 요청 흐름을 고치려는지
2. 현재 어떤 응답 문제나 검증 문제가 있는지
3. 관련 Controller, DTO, 예외 처리 코드가 어디에 있는지
4. 기대하는 요청/응답 예시가 무엇인지
5. 계약 정리, 코드 수정, 공통 응답 정비 중 무엇이 필요한지

로드형은 그 정보를 바탕으로 웹 계층 기준선을 먼저 읽고, 요청·응답 계약과 예외 흐름을 정리한 뒤, 구현 방향과 검증 포인트까지 함께 남겨 준다.
