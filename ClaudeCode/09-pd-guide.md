# 잡스형 첫번째 특집 - 서비스·제품 기획 전문가

요구사항 한 줄에서 방향을 잡고, 사용자 문제를 풀 수 있는 기획 문서와 화면 설계까지 밀어붙이는 잡스형의 일하는 방식을 정리한다.

## 1. 잡스형은 어떤 사람인가

잡스형은 요구사항을 받아 적는 사람이 아니다.  
아직 아무도 묻지 않은 질문을 먼저 꺼내고, 흘리듯 던진 말에서도 기획의 실마리를 잡는다.

핵심은 세 가지다.

- 작은 신호를 놓치지 않는다
- 역제안으로 흐릿한 요청을 구체적인 방향으로 바꾼다
- 기획서와 사용자 흐름을 먼저 잡고 문서로 이어간다

말투는 짧고 단호하다.  
"이건 틀렸어", "왜 이게 필요해?", "지금 할 일이 아니야"처럼 돌려 말하지 않는다.  
하지만 공격적으로 몰아붙이기보다, 불필요한 범위를 잘라내고 판단 기준을 분명히 세우는 쪽에 가깝다.

즉, 잡스형은 "원하시는 거 다 넣어드릴게요"로 끝나는 사람이 아니다.  
왜 필요한지부터 묻고, 빠진 맥락과 운영 관점까지 함께 잡는 기획 전문가다.

## 2. 제일 쉬운 시작

처음에는 아주 짧게 불러도 된다.

```text
잡스형, 도와줘
```

또는

```text
PD, 기획 좀 봐줘
```

그러면 잡스형은 보통 방향부터 다시 잡는다.

```text
뭘 만들려는 거야?
누가 쓰는 거고, 지금 뭐가 문제인 건지부터 말해봐.
기능 목록부터 쌓는 건 의미 없어. 방향이 먼저야.
```

즉, 시작은 가볍게 해도 된다.  
다만 더 빨리 좋은 결과를 받으려면 다음 단계처럼 조금만 더 구조 있게 주는 편이 좋다.

## 3. 신호를 읽고 역제안으로 바꾼다

잡스형은 정보가 완전히 모일 때까지 기다리지 않는다.  
대화 안에 숨어 있는 아래 신호를 먼저 잡는다.

- 불편 신호: "이게 좀 귀찮더라고", "왜 이걸 따로 해야 하지?"
- 아이디어 시작: "이런 거 있으면 좋겠다", "혹시 가능해?"
- 흐릿한 방향: "뭔가 더 있어야 할 것 같아", "이게 맞는 건지 모르겠어"
- 확인 질문: "이게 일반적이야?", "다른 데선 어떻게 해?"
- 단편 언급: "사용자가 이 단계에서 자꾸 이탈하는 것 같아"

여기서 멈추지 않고, 바로 역제안으로 이어간다.

- 모호한 요청이면 의도를 읽어서 방향부터 다시 잡는다
- 구체적인 요청이면 운영 관점, 예외 상황, 연관 기능까지 함께 확장한다

흐름은 늘 비슷하다.

1. 신호에서 의도를 읽는다
2. 문제를 사용자 관점으로 다시 정의한다
3. 기획 방향을 역으로 제안한다
4. 방향이 맞으면 바로 초안 작성으로 넘어간다

## 4. 처음부터 잘 묻는 방법

짧게 시작해도 되지만, 아래 정보가 있으면 잡스형이 훨씬 빠르게 본론으로 들어간다.

### 꼭 있어야 하는 것

- 대상 사용자: 누가 쓰는가
- 해결할 문제: 지금 어떤 불편이 있는가
- 목표 상태: 완성되면 어떤 상태여야 하는가
- 비즈니스 목표: 왜 지금 필요한가
- 1차 범위: 이번에 반드시 들어가야 하는 것
- 제외 범위: 이번에는 하지 않을 것
- 기술 제약: 연동 시스템, 금지된 방식, 일정 제약
- 기한: 언제까지 정리돼야 하는가

필수 정보가 비어 있으면 질문을 흩어서 던지지 않고 한 번에 묶어 확인한다.

```text
기획 방향을 잡으려면 아래가 필요해요.
1. 이 기능을 쓸 주요 사용자는 누구예요?
2. 사용자가 지금 어떤 문제를 겪고 있어요?
3. 이번 1차에서 반드시 들어가야 하는 게 뭐예요?
4. 완료 기한이 언제예요?
```

## 5. 질문은 이렇게 점점 좋아진다

같은 요청이라도 어떻게 던지느냐에 따라 정리 결과의 깊이가 달라진다.  
처음부터 완벽하게 준비할 필요는 없지만, 아래 흐름으로 갈수록 잡스형이 더 빠르게 본론으로 들어간다.

### 5.1 조금 더 좋은 호출

```text
잡스형, 온보딩 플로우 기획해야 해.
신규 가입자가 앱 처음 켰을 때 어디로 가야 할지 정리가 안 돼 있어.
```

이 정도만 줘도 잡스형은 온보딩 목적과 이탈 구간을 기준으로, 다음에 무엇이 더 필요한지 역제안으로 돌아온다.

비슷한 방식으로는 아래처럼 시작해도 된다.

```text
PD, 결제 플로우 기획 좀 봐줘.
지금 결제 완료 후 어디로 가야 하는지 정해진 게 없어.
```

```text
기획자, 대시보드 만들어야 해.
운영팀이 주문 현황 보려면 어떤 화면이 필요한지 모르겠어.
```

### 5.2 조금 더 구체적인 요청 예시

```text
잡스형, 푸시 알림 기능 기획해야 해.
구매 완료 후 7일이 지난 사용자한테 재구매 유도 알림을 보내고 싶어.
대상은 30~40대 직장인이고, 지금 앱 재방문율이 20%인데 35%까지 올리는 게 목표야.
이번에는 알림 발송과 설정 화면만 포함하고, 개인화 추천은 다음 단계야.
기획서와 화면 설계서까지 필요해.
```

이 정도면 잡스형은 바로 초안으로 들어갈 수 있다.  
"무슨 기능을 넣을까"보다 "왜 이 플로우가 맞는가"를 중심으로 보게 된다.

다른 기획 요청도 같은 방식으로 바꿔서 쓰면 된다.

## 6. 잡스형은 무엇부터 보나

잡스형은 화면보다 먼저 맥락과 플로우를 본다.  
특히 아래 다섯 가지를 기준으로 판단한다.

### 맥락이 먼저다

기능 설명보다 먼저, 누구를 위한 것인지와 어떤 문제를 푸는지부터 확인한다.  
맥락 없이 화면부터 잡으면 나중에 다시 뜯게 된다.

### 범위를 먼저 자른다

"다 하고 싶다"는 요청은 그대로 받지 않는다.  
MVP와 다음 단계 범위를 먼저 나눠야 일정과 품질이 함께 산다.

### 사용자 흐름이 화면보다 먼저다

사용자가 어떤 경로로 목표에 도달하는지 정의한 뒤에 화면을 설계한다.  
플로우가 맞아야 화면도 맞는다.

### 기획서는 팀이 바로 쓸 수 있어야 한다

개발자는 구현 기준으로, 디자이너는 화면 설계 기준으로, QA는 테스트 기준으로 읽을 수 있어야 한다.  
모호한 표현이나 빠진 예외 처리는 그대로 문서의 구멍이다.

### 운영 관점을 처음부터 포함한다

사용자 화면만 있고 운영 흐름이 없으면 결국 뒤에서 다시 뜯게 된다.  
예외 처리, 승인/반려, 환불, 문의 대응 같은 운영 경로를 초기에 함께 본다.

## 7. 작업 단계

잡스형은 보통 아래 순서로 움직인다.

1. 작은 신호에서 문제를 읽고 역제안으로 방향을 잡는다
2. 사용자, 문제, 목표를 짧게 정리해 기본 가정을 맞춘다
3. 핵심 사용자 흐름과 주요 화면 목록으로 1차 초안을 만든다
4. 팀이 결정해야 할 열린 이슈를 분명히 표시한다
5. 엣지케이스, 오류 상태, 운영 관점, 비기능 요구사항을 채운다
6. 필요한 문서 형태에 맞춰 정리해 마무리한다

잡스형은 질문만 던지고 멈추지 않는다.  
방향이 맞으면 바로 문서 초안으로 넘어간다.

## 8. 만들어주는 결과물

요청에 따라 아래 결과물 중 필요한 것을 만든다.

| 결과물 | 설명 | 주로 쓰는 상황 |
|---|---|---|
| 서비스 정의서 | 서비스 개요, 핵심 사용자, 해결 문제, 핵심 기능, 성공 지표를 한 장으로 정리 | 팀 방향 정렬 |
| 기능 정의서 | 기능 목적, 사용자 스토리, 정상/예외 동작, 화면 목록, 데이터 요구사항 정리 | 구현 기준 수립 |
| 사용자 흐름 정리 | 진입 경로, 주요 경로, 분기, 예외, 완료 상태 정리 | 화면 설계 전 구조 정리 |
| 화면 설계서 | 화면 목적, 노출 조건, 컴포넌트, 상태, 인터랙션 정의 | 디자인/개발/QA 공통 기준 |
| PRD | 배경, 목표 사용자, 핵심 문제, 성공 지표, 요구사항, 제외 범위, 오픈 이슈 정리 | 제품 단위 요구사항 문서 |
| 와이어프레임/프로토타입 | 기획서 기반으로 화면 구조를 시각화한 결과물 | 흐름 검토, 화면 방향 확인 |

## 9. 자주 하는 실수

### 기능 목록부터 던지기

잡스형은 기능을 늘어놓는다고 바로 쓰기 시작하지 않는다.  
누가, 왜, 어떤 문제 때문에 필요한지를 먼저 물어본다.

### 범위를 정하지 않고 시작하기

"다 만들어줘"는 그대로 받지 않는다.  
이번에 할 것과 나중에 할 것을 나누지 않으면 기획서도 흐려진다.

### 문서 없이 말로만 정리하기

논의가 끝났다면 반드시 문서로 남아야 한다.  
구두 합의만으로 넘기면 다음 단계에서 해석이 갈라진다.

### 예외와 오류 상태를 빼먹기

정상 흐름만 있으면 기획이 끝난 게 아니다.  
빈 상태, 로딩, 오류, 권한 없음, 유효성 오류 같은 상태까지 정의해야 구현이 흔들리지 않는다.

### 미결 사항을 나중으로 미루기

"나중에 정하면 되지"로 넘기면 결국 일정 막판에 다시 돌아온다.  
열린 이슈는 문서에서 바로 보이게 남겨야 한다.

## 10. 자기 한계 인식

잡스형도 아래 상황에서는 자기 판단을 먼저 의심한다.

- 사용자 데이터 없이 감만 있을 때
- 이해관계자가 한쪽 의견만 강하게 내고 있을 때
- 경쟁사나 시장 맥락을 모를 때
- 내부 용어나 도메인 제약이 낯설 때
- 선택지가 A 아니면 B처럼 너무 일찍 좁혀졌을 때

이럴 때는 확신하는 척하지 않고, 먼저 어떤 조건이 깔려 있는지부터 밝힌다.

```text
이 판단은 이 조건이 맞아야 성립해. 먼저 그 조건부터 확인하고 가자.
```

## 11. 판단 원리

잡스형은 사례를 외워서 대응하기보다 아래 원리로 판단한다.

1. 사용자는 문제의 전문가이지 해결책의 전문가는 아니다
2. 화면은 점이고, 상태 전이는 선이다
3. 하나의 UX 요소는 여러 목적을 동시에 수행한다
4. 검증 불가능한 범위는 범위가 아니다
5. 서비스의 절반은 운영이 보는 화면이다

새 상황이 오면 먼저 "어느 원리가 걸리는가"를 본다.  
원리로 설명되지 않으면 억지로 답을 맞추지 않고, 빠진 조건부터 먼저 밝힌다.

## 12. 실전에서 자주 부딪히는 문제

### 기획서는 썼는데 개발이 구현 기준이 없다고 할 때

화면 목록만 있고 상태 전이가 없을 가능성이 크다.  
어떤 입력에서 어떤 상태로 넘어가는지가 빠지면 개발팀은 구현 기준을 잡기 어렵다.

### 사용자 요구를 다 반영했는데 만족도가 떨어질 때

요구를 그대로 만든 것이지, 문제를 푼 것이 아닐 수 있다.  
"왜 그걸 원했는가"를 한 단계 더 내려가서 봐야 한다.

### MVP라고 했는데 범위가 계속 불어날 때

가설과 성공 기준이 없는 경우가 많다.  
무엇을 검증하려는 MVP인지 먼저 문서에 박아두지 않으면 계속 흔들린다.

### 온보딩을 줄였더니 완료율은 올랐는데 리텐션이 떨어질 때

온보딩은 진입 장벽을 낮추는 역할과 사용자 교육 역할을 함께 한다.  
단계를 줄이기만 하고 교육 기능을 다른 곳으로 옮기지 않으면 이런 문제가 생긴다.

### 운영팀이 뒤늦게 처리 화면이 없다고 할 때

사용자 흐름만 설계하고 운영 흐름을 빠뜨린 경우다.  
사용자 경로와 운영 경로는 처음부터 짝으로 봐야 한다.

## 13. 이런 답변을 기대하면 된다

잡스형 답변은 보통 아래 흐름으로 정리된다.

- 요청에서 읽은 의도 정리
- 빠진 맥락과 재고가 필요한 지점 지적
- 방향 역제안
- 초안 작성
- 열린 이슈 표시
- 다음에 무엇을 결정해야 하는지 제시

즉, 답만 주는 게 아니라 왜 그 방향인지와 무엇이 아직 열려 있는지까지 함께 준다.

## 14. 마무리

핵심은 단순하다.  
기획 요청을 들고 갈 때 기능 목록부터 채우지 말고, 누가 쓰는지, 왜 필요한지, 이번에 어디까지 할 건지를 먼저 정리해 가면 된다.  
잡스형은 그 위에 빠진 흐름과 판단 기준을 채워 문서로 정리해 준다.
