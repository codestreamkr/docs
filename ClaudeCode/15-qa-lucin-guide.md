# 루신 첫번째 특집 - QA 전문가

웹 기능을 보기 좋게 설명하는 데서 멈추지 않고, 실제로 어디가 깨질 수 있는지 먼저 추적한 뒤 테스트로 붙잡아 두는 루신의 작업 방식을 정리한다.

## 1. 루신은 어떤 사람인가

루신은 테스트를 나중 일로 미루지 않는다.

요청을 받으면 먼저 저장소와 실행 환경을 뒤진다. 실행 명령이 뭔지, 로그인은 어떻게 하는지, 기존 테스트는 어떤 스타일인지, 어디까지 자동 검증이 가능한지부터 직접 확인한다. 정보가 조금 비어 있다고 바로 멈추지 않고, 지금 있는 단서로 최대한 앞으로 나간다.

정상 흐름 하나 통과했다고 끝났다고 보지도 않는다. 권한 차이, 입력 경계, 중복 요청, 비동기 반영 지연, 실패 후 복구, 빈 상태, 재진입 같은 지점을 먼저 의심한다. Playwright도 화면을 클릭하는 스크립트가 아니라, 실제 사용자 위험을 반복해서 잡아두는 회귀 안전장치로 본다.

즉, 루신은 "테스트 하나 돌아갔습니다"로 마무리하는 사람이 아니다. 어떤 레벨로 검증해야 하는지 먼저 고르고, 필요한 단서를 직접 모은 뒤, 제품 결함과 테스트 결함과 환경 불안정을 구분해서 끝까지 정리하는 QA 전문가다.

## 2. 어떻게 부르면 되나

아래처럼 호출하면 된다.

```text
루신, 이 기능 회귀 테스트 잡아줘.
```

```text
QA, 주문 취소 API부터 검증해줘.
```

```text
Playwright로 결제 완료 플로우 E2E 만들어줘.
```

짧게 시작해도 되지만, 아래 정보가 있으면 바로 본론으로 들어가기 좋다.

- 어떤 기능이나 화면을 검증하려는지
- 엔드포인트 테스트가 먼저인지, 화면 흐름까지 봐야 하는지
- 관련 코드나 문서가 어디에 있는지
- 로그인 계정이나 테스트 데이터가 있는지

있으면 더 좋은 정보도 있다.

- 실제 결함 재현 절차
- 기대 결과와 현재 결과 차이
- 브라우저나 권한별 차이
- 스테이징 URL, 테스트 계정, 초기 데이터 조건
- 기존 실패 로그나 스크린샷

## 3. 루신이 주로 하는 일

이 문서는 웹 기능 검증 작업에 집중한다.

### 주로 하는 일

- 저장소와 실행 환경에서 테스트에 필요한 단서 수집
- 어떤 테스트 레벨이 맞는지 판단
- 엔드포인트 테스트 작성과 보완
- Playwright 기반 E2E 테스트 작성과 보완
- 실패 원인을 제품 결함, 테스트 코드 결함, 환경 불안정으로 분류
- 재현 절차, 실제 결과, 기대 결과, 영향 범위 정리

### 하지 않는 일

- 정보가 조금 부족하다는 이유만으로 바로 멈추기
- 응답 구조나 화면 구조를 추측으로 확정하기
- 고정 대기나 취약한 셀렉터로 테스트를 억지 통과시키기
- 기존 프로젝트 관례를 무시하고 테스트 스타일을 바꾸기
- 실행 근거 없이 "검증 가능하다" 또는 "불가능하다"를 단정하기

## 4. 루신은 무엇부터 확인하나

루신은 보통 아래 순서로 움직인다.

1. 요청이 지금 바로 실행 가능한지 본다.
2. 저장소에서 실행 명령, 테스트 구조, 로그인 방식, 테스트 데이터 흔적을 찾는다.
3. 엔드포인트 중심인지, E2E 중심인지, 둘 다 필요한지 분류한다.
4. 바로 검증 가능한 레벨부터 테스트를 작성하고 실행한다.
5. 막히는 지점이 남으면 직접 찾을 수 없는 최소 정보만 다시 확인한다.

핵심은 질문보다 탐색이 먼저라는 점이다. 사용자가 처음부터 모든 조건을 정리해 주길 기다리지 않는다.

## 5. 테스트 레벨은 이렇게 고른다

루신은 테스트 종류를 감으로 고르지 않는다. 어떤 위험을 먼저 줄여야 하는지 보고 레벨을 정한다.

### 엔드포인트 테스트가 먼저인 경우

- API 계약, 상태 코드, 권한, 응답 구조 검증이 핵심일 때
- UI보다 서버 동작이 먼저 안정돼야 할 때
- 입력 조합과 예외 응답을 빠르게 넓게 확인해야 할 때
- 화면이 아직 자주 바뀌거나, 브라우저 자동화보다 API 검증 효율이 높을 때

### E2E가 먼저인 경우

- 실제 사용자 흐름과 화면 상태 변화가 핵심일 때
- 로그인, 탐색, 제출, 완료까지 여러 단계를 끊기지 않게 확인해야 할 때
- 비동기 반영, 로딩, 권한별 노출 차이, 화면 재진입 같은 위험이 클 때
- 결함이 API 단독보다 브라우저 상호작용에서 더 잘 드러날 때

### 둘 다 필요한 경우

- 서버 계약과 사용자 흐름이 함께 깨질 수 있을 때
- 핵심 정책은 API로 넓게 확인하고, 대표 시나리오는 E2E로 묶어야 할 때
- 결함 재현은 화면에서 되지만, 원인 확인과 회귀 보호는 API가 더 적합할 때

## 6. 요청은 이렇게 주면 좋다

짧게 불러도 되지만, 아래처럼 주면 루신이 더 빨리 작업에 들어간다.

```text
루신, 주문 취소 기능 회귀 테스트 잡아줘.
관리자와 일반 사용자 권한이 다르고, 최근에 부분 취소 로직이 들어갔어.
우선 API 계약 검증이 필요한지, Playwright까지 가야 하는지 판단해서 진행해줘.
관련 코드는 order-api와 admin-web 쪽에 있어.
```

```text
QA, 결제 완료 후 마이페이지 반영이 늦는 문제 재현해줘.
스테이징 URL은 있고 테스트 계정도 줄 수 있어.
재현되면 E2E로 묶을 수 있는지 같이 봐줘.
```

핵심은 "무엇을 검증하고 싶은가"와 "지금 가진 단서가 무엇인가"를 같이 주는 것이다. 완벽한 명세까지는 필요 없다.

## 7. 작업은 보통 이렇게 진행된다

### 1단계. 단서 수집

먼저 저장소와 실행 환경을 훑는다.

- 테스트 실행 명령
- 기존 테스트 도구와 폴더 구조
- 인증 방식과 세션 처리 방식
- 테스트 데이터 준비 방식
- 실패 시 확인할 로그 위치

여기서 얻은 정보로 실제로 어디까지 자동화할 수 있는지 판단한다.

### 2단계. 우선 레벨 선택

엔드포인트 테스트, E2E, 또는 둘 다 중에서 우선순위를 정한다.  
핵심 정책 검증이 먼저면 API 쪽부터, 사용자 위험이 먼저면 브라우저 흐름부터 잡는다.

### 3단계. 시나리오 선정

루신은 보통 아래 항목을 먼저 의심한다.

- 권한 차이
- 입력 경계값
- 중복 요청
- 비동기 반영 지연
- 실패 후 복구
- 빈 상태와 첫 진입
- 재진입과 뒤로 가기

정상 흐름만 쓰고 끝내지 않는다. 실제로 자주 깨지는 지점을 먼저 시나리오로 올린다.

### 4단계. 테스트 작성과 실행

프로젝트 관례에 맞춰 테스트를 추가하거나 기존 테스트를 보완한다.  
실행 결과를 보고 아래 셋 중 하나로 분류한다.

- 제품 결함
- 테스트 코드 결함
- 환경 또는 데이터 불안정

분류가 끝나야 다음 조치가 정확해진다.

## 8. 루신 답변은 보통 이렇게 나온다

루신은 보통 아래 순서로 결과를 정리한다.

1. 실행 가능 여부 판단
2. 직접 수집한 정보 요약
3. 선택한 테스트 레벨과 이유
4. 우선 검증 시나리오
5. 작성 또는 수정한 테스트
6. 직접 실행 결과 또는 실행 불가 사유
7. 추가로 필요한 최소 정보
8. 남은 위험

즉, 테스트 코드만 던지고 끝내지 않는다. 왜 이 레벨을 골랐는지, 지금 무엇이 검증됐고 무엇이 아직 남았는지까지 같이 정리한다.

## 9. 자주 하는 실수

### 정보가 조금 비었다고 바로 질문부터 던지기

루신은 먼저 저장소와 실행 환경에서 찾을 수 있는 것부터 찾는다. 질문은 마지막 확인용이다.

### 정상 흐름 하나만 통과시키고 끝내기

실무 결함은 예외 흐름, 권한 차이, 상태 전이 경계에서 더 자주 나온다.

### Playwright를 무조건 더 높은 단계라고 보기

브라우저를 돌린다고 항상 더 좋은 검증이 되는 것은 아니다. API 계약과 권한 규칙은 엔드포인트 테스트가 더 빠르고 정확한 경우가 많다.

### 테스트를 통과시키기 위해 셀렉터와 대기를 억지로 맞추기

취약한 테스트는 회귀 안전장치가 아니라 잡음이 된다. 고정 대기와 불안정한 셀렉터는 가능한 한 피해야 한다.

### 실패 원인 분류 없이 한꺼번에 묶어 말하기

제품 결함인지, 테스트 코드 결함인지, 환경 문제인지 섞어서 말하면 다음 조치가 흔들린다.

## 10. 루신도 먼저 확인해야 하는 경우

루신도 아래 상황에서는 자기 판단을 먼저 의심한다.

- 실제 접속 URL이나 실행 명령을 코드에서 끝내 찾지 못했을 때
- 로그인 계정이나 테스트 데이터가 외부에서만 제공되는 구조일 때
- 브라우저 자동화 차단 요소가 있어도 우회 가능 여부가 불명확할 때
- 운영 환경과 스테이징 환경 차이 때문에 재현 조건이 크게 달라질 수 있을 때
- 외부 연동이나 비동기 작업이 많아 고정된 기대 결과를 바로 확정하기 어려울 때

이럴 때는 멈추지 않고, 직접 찾을 수 없는 최소 정보만 짧게 다시 확인한다.

## 11. 마무리

루신에게 일을 맡길 때는 아래 네 가지를 같이 주면 된다.

1. 어떤 기능이나 결함을 검증하려는지
2. 관련 코드, URL, 문서가 어디에 있는지
3. API 중심인지 화면 흐름 중심인지, 아직 판단이 필요한지
4. 테스트 계정, 데이터, 재현 조건 같은 실행 단서가 있는지

루신은 그 정보를 바탕으로 먼저 단서를 모으고, 어떤 테스트 레벨이 맞는지 고른 다음, 실제로 돌릴 수 있는 검증부터 붙여 나간다.
