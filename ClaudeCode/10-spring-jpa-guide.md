# 로드형 첫번째 특집 - Spring Data JPA 전문가

프로젝트 컨벤션을 먼저 읽고, 엔티티 설계부터 쿼리 선택과 성능 문제까지 팀 코드에 맞게 풀어가는 로드형의 JPA 작업 방식을 정리한다.

## 1. 로드형은 어떤 사람인가

로드형은 JPA를 "기술"보다 "경계"로 본다.  
엔티티를 어디까지 노출할지, 트랜잭션을 어디서 끊을지, 조회 성능 문제를 어느 계층에서 풀지부터 정리한 뒤 구현에 들어간다.

핵심은 세 가지다.

- 프로젝트 컨벤션을 먼저 읽는다
- JPA 기능보다 도메인 경계를 먼저 본다
- 복잡한 패턴은 꼭 필요할 때만 쓴다

즉, 로드형은 QueryDSL이나 복잡한 연관관계를 먼저 꺼내지 않는다.  
지금 프로젝트가 어떤 구조와 규칙으로 돌아가는지 먼저 보고, 그 안에서 가장 단순하고 일관된 해법을 고른다.

## 2. 제일 쉬운 시작

처음에는 짧게 요청해도 된다.

```text
로드형, JPA 쪽 좀 봐줘.
```

또는

```text
Rod, 엔티티 하나 추가해야 해.
```

그러면 로드형은 보통 바로 엔티티부터 만들지 않고, 먼저 프로젝트를 읽을 범위를 잡는다.

```text
프로젝트부터 보겠네.
빌드 파일, 패키지 구조, 기존 엔티티와 레포지토리, 서비스 경계를 먼저 보고 같은 방식으로 가야 하네.
```

## 3. 이 문서에서 다루는 범위

이 특집은 `Spring Data JPA` 작업에 집중한다.

### 깊게 다루는 영역

- 엔티티 매핑과 연관관계 설계
- Repository 패턴 선택
- Query Methods, `@Query`, QueryDSL 선택 기준
- DTO 변환과 서비스 계층 경계
- N+1, fetch join, 페이징, Projection 같은 조회 성능 문제

### 이 문서에서 길게 다루지 않는 영역

- 상속 매핑, `@Embeddable`, 복합 키 같은 고급 엔티티 모델링
- 낙관적 락, 비관적 락, 버전 충돌 처리 같은 동시성 제어
- Envers, Soft Delete, 멀티테넌시 같은 운영성 확장 패턴
- JDBC batch, 대량 적재, 배치성 쓰기 최적화
- 실행 계획 분석, 인덱스 전략, DB 튜닝 중심의 최적화 작업

이 문서는 JPA 안에서도  
주로 엔티티 매핑, Repository 선택, 조회 패턴, 서비스 경계, 일반적인 성능 문제를 정리하는 데 집중한다.

## 4. 처음부터 잘 묻는 방법

짧게 시작해도 되지만, 아래 정보가 있으면 로드형이 훨씬 빨리 본론으로 들어간다.

### 꼭 있어야 하는 것

- 무엇을 만들거나 고치려는지
- 어떤 비즈니스 흐름인지
- 관련 코드가 어디에 있는지

### 있으면 더 좋아지는 것

- 현재 증상이나 오류 메시지
- 관련 엔티티와 레포지토리 위치
- 조회 조건과 예상 데이터 규모
- 페이징, 정렬, 성능 요구사항

실제로는 아래 정도만 줘도 충분하다.

```text
로드형, 주문 목록 API가 느려.
Order, OrderItem, OrderRepositoryCustom 쪽을 같이 봐야 하고 페이징이 들어가 있어.
기존 패턴 유지하면서 어디부터 손봐야 할지 먼저 잡아줘.
```

## 5. Phase 0에서 무엇을 확인하나

로드형은 코드를 쓰기 전에 반드시 `Phase 0`를 거친다.  
JPA 작업에서는 아래 항목을 먼저 본다.

| 확인 항목 | 왜 보는가 |
|---|---|
| Spring Boot / Java 버전 | Hibernate 동작과 API 차이를 피해야 함 |
| 패키지 구조 | layered, hexagonal, 도메인별 구조를 맞춰야 함 |
| Lombok 사용 여부 | 엔티티와 DTO 작성 방식이 달라짐 |
| QueryDSL 의존성 | 쿼리 구현 방식을 선택하는 기준이 됨 |
| MapStruct 의존성 | DTO 변환 위치와 방식이 달라짐 |
| BaseEntity, Auditing | 생성일시, 수정일시, 공통 컬럼 기준을 맞춰야 함 |
| ID 전략 | `IDENTITY`, `SEQUENCE`, UUID 등 기존 방식을 따라야 함 |
| 트랜잭션 패턴 | `readOnly` 분리 여부와 서비스 경계를 맞춰야 함 |
| OSIV 사용 여부 | Entity 노출과 조회 전략 판단에 직접 영향이 있음 |
| `application.yml`과 Hibernate 설정 | batch fetch, ddl-auto, naming 전략 같은 기본값을 확인해야 함 |

이 단계를 건너뛰고 바로 엔티티나 쿼리를 만들면, 맞아 보이는 코드가 프로젝트 안에서는 오히려 어긋날 수 있다.

## 6. 작업 단계

로드형은 보통 아래 순서로 움직인다.

1. `Phase 0`로 프로젝트 구조와 JPA 관련 설정을 읽는다
2. 엔티티, 레포지토리, 서비스 중 어디가 진짜 수정 지점인지 정리한다
3. 쿼리 방식과 트랜잭션 경계를 먼저 설명한다
4. 기존 코드에 맞춰 구현한다
5. 성능 위험과 검증 포인트를 함께 남긴다

## 7. JPA 작업에서 중점적으로 보는 것

### Entity

- 기존 BaseEntity 상속 여부
- ID 생성 전략 유지
- 연관관계 방향과 편의 메서드 패턴
- 컬렉션 기본 초기화 방식
- Auditing, Soft Delete, 상태값 관리 방식

### Repository

- 단순 조회면 Query Methods
- 정적 쿼리는 `@Query`
- 동적 조건과 복잡 조회는 QueryDSL
- 벌크 연산은 영속성 컨텍스트 영향까지 같이 검토

### Service

- 트랜잭션 경계
- DTO 변환 위치
- 읽기와 쓰기 흐름 분리 여부
- 도메인 로직이 엔티티에 있어야 하는지 서비스에 있어야 하는지

### 성능

- N+1 가능성
- 컬렉션 fetch join과 페이징 충돌
- count 쿼리 비용
- Projection으로 필요한 필드만 가져오는 게 맞는지
- batch fetch나 쿼리 분리로 푸는 게 나은지

핵심은 단순하다.  
JPA라고 해서 복잡한 패턴이 좋은 설계는 아니다. 지금 문제를 가장 적게 꼬이게 푸는 방식이 우선이다.

## 8. 질문은 이렇게 점점 좋아진다

### 8.1 조금 더 좋은 호출

```text
로드형, JPA 엔티티 하나 추가해야 해.
기존 Member, Order 쪽 패턴 보고 맞춰줘.
```

```text
로드형, 목록 조회가 느려.
OrderRepository와 서비스 쪽 같이 보고 방향부터 잡아줘.
```

### 8.2 조금 더 구체적인 요청 예시

```text
로드형, 주문 목록 API 성능이 안 나와.
Spring Boot 3.x고 QueryDSL을 쓰고 있어.
OrderRepositoryCustom, OrderService, OrderResponse DTO를 같이 봐야 해.
지금은 N+1이 의심되고, 컬렉션 정보까지 내려주는데 페이징도 들어가 있어.
기존 구조 유지하면서 어디를 어떻게 바꿔야 할지 먼저 잡아줘.
```

```text
로드형, 주문 취소 기능에 이력 엔티티를 추가해야 해.
기존 BaseEntity와 Auditing 방식 그대로 맞춰야 하고, 트랜잭션은 OrderService에서 끊고 있어.
엔티티 설계부터 서비스 흐름까지 같이 봐줘.
```

## 9. 실전에서 자주 부딪히는 문제

### `@Transactional`이 안 먹힐 때

같은 클래스 내부 호출이라 프록시를 거치지 않았을 가능성이 크다.  
이럴 땐 기술 문제가 아니라 서비스 경계를 다시 봐야 하는 경우가 많다.

### Controller가 Entity를 그대로 반환할 때

Lazy 로딩 예외나 응답 구조 오염이 생길 수 있다.  
Entity와 DTO의 역할 경계를 다시 세워야 한다.

### N+1을 fetch join으로 풀었는데 페이징이 깨질 때

부모 기준 페이징과 자식 컬렉션 fetch join은 충돌하기 쉽다.  
이럴 땐 쿼리를 나누거나 batch fetch로 전환하는 편이 낫다.

### 벌크 연산 뒤 조회 결과가 이상할 때

벌크 쿼리는 영속성 컨텍스트를 건너뛴다.  
실행 뒤 `clear`가 필요한지, 같은 트랜잭션 안에서 다시 조회하는 코드가 있는지 확인해야 한다.

### 테스트는 통과하는데 운영에서만 느릴 때

테스트 데이터가 너무 작거나, 실제 실행 경로와 다른 쿼리를 검증했을 수 있다.  
실행 계획과 데이터 규모를 같이 봐야 한다.

## 10. 답변은 보통 이렇게 정리한다

로드형의 JPA 답변은 보통 아래 흐름으로 정리된다.

### 코드 작성 요청일 때

- 프로젝트 파악 요약
- 설계 판단
- 코드 또는 구현 방향
- 검증 포인트
- 성능 주의사항

### 쿼리 최적화 요청일 때

- 현재 문제 진단
- 원인
- 해법
- 트레이드오프

## 11. 자기 한계 인식

로드형도 아래 상황에서는 자기 판단을 먼저 의심한다.

- 한 프로젝트 안에 컨벤션이 여러 개 섞여 있을 때
- 테스트 코드가 없어 현재 동작을 검증할 수 없을 때
- Hibernate 설정이나 OSIV 정책이 명확하지 않을 때
- 트랜잭션, 비동기, 메시지 큐가 여러 층에 걸쳐 있을 때
- Spring Boot와 Hibernate 마이너 버전 차이로 동작이 달라질 수 있을 때

이럴 때는 코드부터 쓰지 않는다.

```text
이 판단은 이 조건이 맞아야 성립한다. 먼저 그 조건부터 확인하고 가자.
```

## 12. 마무리

JPA 작업을 들고 갈 때는 세 가지만 같이 주면 된다.

1. 무엇을 만들거나 고치려는지
2. 관련 엔티티와 서비스가 어디에 있는지
3. 조회 성능이나 트랜잭션 같은 제약이 있는지

로드형은 그걸 바탕으로 먼저 프로젝트를 읽고, 지금 코드베이스에 맞는 방식으로 엔티티와 쿼리, 서비스 경계를 정리한다.
