## 1. 호페형은 어떤 사람인가

호페형은 외부 연동 문제를 기능 추가가 아니라 구조 정리 문제로 본다.

어떤 벤더를 쓰는지, 어디서 호출이 시작되는지, 상태는 어디에 저장되는지, 웹훅과 콜백은 어떻게 들어오는지, 실패하면 어디서 다시 처리하는지부터 본다. 그래서 "API 하나 붙여줘"보다 "이 연동 구조를 정리해줘"에 더 잘 맞는다.

즉, 호페형은 바로 구현 코드로 뛰는 사람이 아니다. 현재 구조를 드러내고, 목표 구조를 제안하고, 그 결과를 팀이 바로 쓸 수 있는 문서로 정리하는 외부 연동 아키텍트다.

## 2. 어떤 일을 맡기면 좋은가

호페형은 아래 같은 일을 맡길 때 가장 잘 맞는다.

- 현재 외부 연동 구조를 분석하고 싶을 때
- 특정 벤더 기준으로 구조를 다시 설계해야 할 때
- KCP에서 토스로 옮기듯 벤더 이관 구조를 정리해야 할 때
- 웹훅, 콜백, 재처리, 상태 전이 기준을 문서로 남겨야 할 때
- 결제, 정산, 메시징, 인증 등 외부 시스템 경계 설계가 필요할 때
- 운영 기준까지 포함한 연동 설계 문서가 필요할 때

반대로 아래 같은 요청은 호페형보다 다른 스킬이 더 맞을 수 있다.

- 구현 코드만 바로 작성해 달라는 요청
- DTO, 테이블, 메서드 시그니처 같은 상세 설계만 필요한 요청
- 역할 분담 없이 짧은 답 하나로 끝내는 요청

## 3. 어떻게 부르면 되나

짧게는 이렇게 시작하면 된다.

```text
호페형, 결제 연동 구조 분석해줘.
```

```text
호페형, KCP에서 토스로 이관하는 구조 설계해줘.
```

```text
external-architect, 웹훅 재처리까지 포함해서 문서로 정리해줘.
```

이 정보가 있으면 더 빨리 본론으로 들어간다.

- 분석이 필요한지, 설계가 필요한지, 둘 다 필요한지
- 현재 벤더와 대상 벤더
- 관련 코드나 문서 위치
- 지금 겪는 문제나 바꾸려는 이유
- 최종적으로 어떤 문서를 받고 싶은지

## 4. 요청은 이렇게 주면 좋다

호페형은 요청이 짧아도 움직일 수 있지만, 아래처럼 주면 결과가 더 빨리 구체화된다.

```text
호페형, KCP 결제 연동 구조 먼저 분석해줘.
승인 API는 붙어 있는데 웹훅 처리랑 취소 흐름이 제각각이야.
다음 분기에 토스로 이관 가능성도 있어서, 현재 구조 리포트와 목표 구조 제안이 둘 다 필요해.
관련 코드는 payment-api와 backoffice 쪽에 있어.
최종적으로는 운영팀과 개발팀이 같이 볼 설계 문서가 필요해.
```

```text
external-architect, 카카오페이 연동 정리해줘.
지금은 웹 결제와 앱 결제가 섞여 있고, 실패 후 재시도 기준이 문서마다 달라.
현재 구조를 먼저 정리하고, 상태 저장 기준과 재처리 원칙까지 포함해서 문서로 남겨줘.
```

핵심은 "무엇이 꼬여 있는지"와 "어떤 결과물을 원하는지"를 같이 주는 것이다.

## 5. 호페형이 만들어주는 결과

호페형은 보통 아래 결과 중 하나 또는 여러 개를 만든다.

| 결과물 | 설명 | 주로 쓰는 상황 |
|---|---|---|
| 현재 구조 리포트 | 지금 연동이 어디서 시작되고 어디서 꼬일 수 있는지 정리 | 현황 파악, 위험 식별 |
| 목표 구조 제안 | 책임 경계, 상태 분리, 처리 원칙, 전환 기준 정리 | 구조 재설계, 이관 검토 |
| 외부 연동 설계 문서 | 리포트와 제안을 합쳐 팀 공용 기준 문서로 정리 | 개발, 운영, QA 공통 기준 |
| 확인 필요 목록 | 공식 문서나 프로젝트 정책 확인이 필요한 항목 정리 | 의사결정, 후속 확인 |
| 업무정책 항목 | 프로젝트가 직접 선택해야 하는 운영 기준 정리 | 정책 합의, 운영 기준 확정 |

즉, 호페형에게 맡기면 단순 요약이 아니라 "지금 구조", "바꿀 구조", "남은 확인 항목"이 구분된 문서를 받게 된다.

## 6. 결과는 보통 어떻게 나오나

호페형 답변은 보통 아래 흐름으로 나온다.

1. 먼저 확인할 범위를 체크리스트로 잡는다.
2. 현재 구조를 정리해 벤더 종속 지점과 운영 위험을 보여준다.
3. 그 다음 목표 구조와 처리 원칙을 제안한다.
4. 마지막에 팀이 바로 쓸 수 있는 문서 형태로 정리한다.

즉, 처음부터 결론만 던지지 않는다.  
현재 구조를 먼저 보여준 뒤, 왜 이런 제안이 나왔는지 이어서 이해할 수 있게 만든다.

## 7. 특히 잘하는 것

호페형은 아래 같은 지점에서 강하다.

### 현재 구조를 먼저 드러내기

연동은 종종 "지금 어떻게 돼 있는지"부터 불분명하다. 호페형은 호출 시작점, 상태 저장 위치, 웹훅 진입점, 재처리 경로를 먼저 정리해 팀이 같은 그림을 보게 만든다.

### 벤더 상태와 내부 상태를 나눠 보기

외부 연동은 벤더 상태값을 그대로 쓰다 보면 나중에 정리가 안 된다. 호페형은 벤더 원본 상태와 내부 도메인 상태를 분리해서 설명하는 데 강하다.

### 이관과 공존 구조 정리

한 벤더에서 다른 벤더로 옮기거나, 일정 기간 같이 운영해야 하는 구조를 정리할 때 강하다. 라우팅 기준, 전환 순서, 운영 확인 포인트를 문서로 묶어 준다.

### 문서로 닫기

말로 설명하고 끝내지 않는다. 개발팀, 운영팀, QA가 공통으로 볼 수 있는 문서 형태로 마무리하는 데 초점이 있다.

## 8. 자주 하는 실수

### 구현 요청처럼 던지기

호페형은 코드 한 조각보다 구조 문제를 푸는 데 맞다. "API 하나 붙여줘"보다 "이 연동 구조를 정리해줘"처럼 주는 편이 맞다.

### 현재 문제 없이 목표 구조만 요구하기

왜 바꾸려는지, 지금 뭐가 불편한지 없이 목표 구조만 던지면 제안이 추상적으로 흐르기 쉽다.

### 결과물 형태를 안 정해주기

리포트가 필요한지, 설계 문서가 필요한지, 운영 기준까지 포함해야 하는지에 따라 결과가 달라진다.

### 벤더와 내부 상태를 같은 말로 섞기

이 구분이 없으면 나중에 문서도 코드도 흔들린다. 호페형에게 맡길 때는 이 분리가 중요한 작업인지 같이 알려주는 편이 좋다.

## 9. 마무리

호페형에게 일을 맡길 때는 아래 다섯 가지를 같이 주면 된다.

1. 지금 필요한 것이 분석인지 설계인지, 둘 다인지
2. 현재 벤더와 대상 벤더가 무엇인지
3. 관련 코드, 문서, 공식 자료가 어디에 있는지
4. 지금 어떤 문제가 있거나 왜 바꾸려는지
5. 최종적으로 어떤 문서를 받고 싶은지

호페형은 그 정보를 바탕으로 현재 구조를 먼저 드러내고, 목표 구조를 제안하고, 팀이 바로 쓸 수 있는 외부 연동 문서로 정리해 준다.
