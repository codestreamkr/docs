# 06. [중급]프로젝트 코딩

## 이 문서는 언제 보나

프로젝트를 손봐야 하는데 어디서부터 들어가야 할지 애매할 때 본다.

기능 하나를 바로 붙이는 경우도 있고, 리팩토링이나 구조 개선처럼 먼저 판단이 필요한 경우도 있다.

이 문서는 그런 상황에서 `ct-improve-plan`, `ct-design-review`, `ct-implement`를 어떻게 이어서 쓰면 되는지 정리한 개발 가이드다.

문서 안의 코드 블록 예시는 실제 입력 형태에 맞춰 `/ct:implement` 같은 형식으로 적었다.

## 먼저 감을 잡자

### 구조 개선, 리팩토링, 기술부채 정리

- 추천 흐름: `ct-improve-plan` -> `ct-design-review` -> `ct-implement`
- 이유: 무엇을 먼저 바꿀지 판단하고, 설계를 정리한 뒤 구현해야 재작업이 줄어든다.

### 새 기능인데 정책, 권한, 운영 영향이 큰 경우

- 추천 흐름: `ct-design-review` -> `ct-implement`
- 이유: 구현 전에 범위와 운영 기준을 먼저 정리하는 편이 안전하다.

### 단일 기능 추가, 범위가 작고 정책이 단순한 경우

- 추천 흐름: `ct-implement`
- 이유: 문서 단계를 줄이고 바로 구현으로 들어가는 편이 빠르다.

### 이미 개선 계획 문서가 있는 경우

- 추천 흐름: `ct-design-review` -> `ct-implement`
- 이유: 우선순위 판단은 끝났으니 구현 방향과 실제 반영 범위만 정리하면 된다.

## 각 스킬은 무슨 일을 하나

### `ct-improve-plan`

- 하는 일: 코드와 설정을 근거로 개선 과제를 뽑고 우선순위를 정한다.
- 결과물: `{project}_improve_plan.md`

### `ct-design-review`

- 하는 일: 여러 전문가 관점으로 설계안을 검토하고, 충돌을 조율해 최종 채택안을 정리한다.
- 결과물: `{topic}_design_review.md`

### `ct-implement`

- 하는 일: 구현 가능 범위를 먼저 정리한 뒤, 승인되면 실제 코드 수정까지 이어서 처리한다.
- 결과물: 코드 변경 + 영향 범위 정리

핵심은 역할을 섞지 않는 것이다.

- `ct-improve-plan`은 판단 문서 작성용이다. 코드를 바로 고치지 않는다.
- `ct-design-review`는 설계 합의용이다. 구현 전에 전문가 리뷰, 쟁점, 채택안과 제외안을 정리한다.
- `ct-implement`는 실행 단계다. 다만 계획 문서나 설계 검토 문서를 입력으로 받으면 바로 코드를 바꾸지 않고, 먼저 구현 가능 범위를 정리한 뒤 승인을 기다린다.

## 흐름 1. 큰 작업은 3단계로 끊는다

리팩토링, 구조 개선, 운영 안정화, 성능 개선처럼 범위가 넓은 작업은 아래 순서가 기본이다.

### Step 1. `ct-improve-plan`으로 먼저 판을 깐다

무엇이 문제인지보다 무엇부터 손대야 하는지를 먼저 정리한다.

예시:

```text
/ct:improve-plan 이 프로젝트의 개선 과제를 우선순위 기준으로 정리해줘.
```

이 단계에서 기대하는 결과는 아래와 같다.

- 개선 항목이 `IMP-xxx` 형태로 정리된다.
- 각 항목마다 왜 필요한지, 어디까지 손댈지, 바로 위임 가능한지 구분된다.
- 당장 구현해도 되는 항목과 추가 확인이 필요한 항목이 갈린다.

여기서 바로 구현으로 뛰어들지 않는 이유는, 큰 작업일수록 "좋아 보이는 수정"보다 "먼저 고쳐야 할 순서"가 더 중요하기 때문이다.

### Step 2. `ct-design-review`로 구현 방향을 좁힌다

개선 계획에서 바로 구현하면 해석 차이로 다시 돌아오는 경우가 많다.

정책, 운영 모델, 범위 경계가 애매한 항목은 설계 검토를 한 번 거치는 편이 낫다.

이 단계의 중심은 "의견을 많이 모으는 것"이 아니라 "전문가 리뷰를 거쳐 최종안을 확정하는 것"이다.

`ct-design-review`는 보통 아래 흐름으로 읽으면 된다.

- 1차 전문가 리뷰: product, backend, frontend-ux, domain 같은 역할이 각자 판단과 우려를 낸다.
- 후행 검토: qa, security, sre처럼 앞선 설계안을 보고 품질과 운영 리스크를 덧붙인다.
- 최종 조정: tech-lead 관점에서 충돌 지점을 정리하고 채택안, 제외안, 보류안을 확정한다.

예시:

```text
/ct:design-review IMP-002를 기준으로 게시판 기능 구조를 검토해줘.
관리자/일반 사용자 권한 차이와 첨부파일 처리 방식까지 포함해줘.
```

이 단계에서 기대하는 결과는 아래와 같다.

- 어떤 전문가가 어떤 근거로 검토했는지 남는다.
- 의견 충돌 지점과 조율 결과가 정리된다.
- 어떤 안을 채택할지 정리된다.
- 제외하거나 보류할 안이 분리된다.
- 구현 순서와 그 이유가 남는다.
- 성공 기준과 재검토 기준이 함께 정리된다.

특히 권한, 외부 연동, 비동기 처리, 운영 복잡도가 있는 기능은 이 단계를 생략하지 않는 편이 안전하다.

### `ct-design-review` 문서에서 꼭 봐야 할 부분

설계 검토 문서를 받았을 때는 아래 순서로 보는 편이 좋다.

#### 1. 참여 전문가

누가 왜 들어왔는지 먼저 본다.

여기서 product, backend만 있는지, security나 sre까지 들어왔는지에 따라 이 기능의 성격과 리스크가 바로 보인다.

#### 2. 전문가별 의견

이 섹션은 단순 참고용이 아니다.

어떤 역할이 무엇을 우려했고, 어떤 조건이 선행돼야 하는지 확인하는 구간이다.

특히 아래 항목은 구현 전에 꼭 확인하는 편이 좋다.

- 판단
- 우려사항
- 권장안
- 필요한 선행 산출물

#### 3. 충돌 지점과 조율 결과

의견이 갈린 부분이 왜 그렇게 정리됐는지 보는 구간이다.

여기가 빠지면 결론만 남고 판단 이유가 사라져서, 구현 단계에서 다시 논쟁이 열리기 쉽다.

#### 4. 최종 결론

실제로 구현할 때 가장 중요하게 보는 부분이다.

여기에는 최소한 아래 내용이 있어야 한다.

- 채택안
- 제외안 또는 보류안
- 선택 이유
- 성공 기준
- 재검토 기준

`ct-implement`로 넘길 때는 이 `최종 결론`과 아래 `권장 구현 방향`을 사실상 기준 문서로 본다.

#### 5. 권장 구현 방향

최종 결론을 실제 작업 순서로 바꿔 놓은 구간이다.

무엇을 먼저 만들지, 왜 그 순서가 필요한지, 이번 라운드에서 무엇을 빼야 하는지 여기서 정리된다.

### Step 3. `ct-implement`로 실제 반영한다

계획 문서나 설계 검토 문서를 넘기면 `ct-implement`는 먼저 아래 내용을 짧게 정리한다.

- 이번에 바로 구현할 범위
- 가정을 두고 구현 가능한 범위
- 정책 미확정으로 보류할 범위

설계 검토 문서를 입력으로 줄 때는 특히 아래 두 부분을 기준으로 구현 범위를 확정한다고 생각하면 된다.

- `최종 결론`
- `권장 구현 방향`

반대로 `남은 이슈`에 있는 내용은 확정사항이 아니라 미결정 항목으로 보고 분리해야 한다.

그다음 사용자 승인을 기다렸다가 구현으로 넘어간다.

예시:

```text
/ct:implement 방금 만든 design review 기준으로 게시판 기능 구현해줘.
```

이 흐름이 잘 맞는 요청은 아래처럼 표현하면 된다.

```text
1. /ct:improve-plan으로 개선 과제 먼저 정리
2. 그중 게시판 기능은 /ct:design-review로 설계 검토
3. 채택안 기준으로 /ct:implement 진행
```

## 흐름 2. 작은 작업은 바로 구현한다

정책이 단순하고 범위가 분명하면 `ct-implement`로 바로 들어가도 된다.

예시:

```text
/ct:implement 게시판 기능 추가해줘.
목록, 상세, 등록, 수정, 삭제만 이번 범위로 하고
권한은 관리자만 수정/삭제 가능하게 해줘.
```

이 방식이 잘 맞는 경우는 아래와 같다.

- 화면과 API 범위가 분명하다.
- 권한 정책이 단순하다.
- 외부 연동이나 배치가 없다.
- 기존 코드 패턴을 따라가면 된다.

반대로 아래 조건이 보이면 바로 설계 검토로 한 단계 올리는 편이 낫다.

- 권한 정책이 여러 역할로 갈린다.
- 상태 전이나 승인 흐름이 있다.
- 파일 업로드, 알림, 배치, 외부 API가 같이 붙는다.
- 운영 중 장애나 데이터 정합성 이슈가 커질 수 있다.

## 요청을 이렇게 쓰면 결과가 좋아진다

짧게 요청해도 되지만, 아래 네 가지를 붙이면 결과가 훨씬 안정적이다.

- 대상: 게시판, 회원관리, 정산 배치
- 이번 범위: 목록/상세/등록만, 관리자 화면만, API만
- 제외 범위: 첨부파일 제외, 알림 제외, 마이그레이션 제외
- 제약: 기존 응답 형식 유지, 특정 테이블 재사용, 테스트는 이번 라운드 제외

예시:

```text
/ct:implement 게시판 기능 추가해줘.
이번 범위는 목록/상세/등록/수정/삭제고,
첨부파일과 알림은 제외해줘.
기존 응답 형식은 유지하고 관리자만 수정/삭제 가능하게 해줘.
```

## 실무에서 많이 쓰는 조합

### 1. 리팩토링 시작할 때

```text
/ct:improve-plan 이 프로젝트의 서비스 레이어 정리 과제를 우선순위로 정리해줘.
```

### 2. 기능 방향이 엇갈릴 때

```text
/ct:design-review 게시판 기능 설계를 검토해줘.
공지/일반글 구분, 권한, 검색 조건, 운영 관점까지 포함해줘.
```

### 3. 바로 개발 맡길 때

```text
/ct:implement 게시판 기능 추가해줘.
```

### 4. 문서 보고 이어서 개발할 때

```text
/ct:implement IMP-003과 게시판 design review 문서를 기준으로 이번 범위만 구현해줘.
```

## 헷갈리기 쉬운 점

- `ct-improve-plan`은 개선 과제 정리용이지 구현용이 아니다.
- `ct-design-review`는 좋은 의견 모음집이 아니라 채택안과 제외안을 정리하는 문서여야 한다.
- `ct-implement`는 문서를 받았다고 바로 구현하지 않는다. 먼저 실행 가능 범위를 정리하고 승인을 기다린다.
- 큰 작업을 바로 `ct-implement`로 밀어 넣으면, 구현 도중 정책이 다시 열릴 가능성이 높다.

## 빠르게 고르는 기준

1. 무엇부터 손댈지 모르면 `ct-improve-plan`
2. 어떻게 만들지 합의가 필요하면 `ct-design-review`
3. 범위가 확정됐고 바로 반영하면 되면 `ct-implement`

이 세 가지를 순서대로 쓰면, 큰 작업도 문서와 코드가 따로 놀지 않고 자연스럽게 이어진다.
