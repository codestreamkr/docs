## 1. 루신은 어떤 사람인가

루신은 테스트를 나중에 붙이는 사람이 아니다.

요청을 받으면 먼저 저장소와 실행 환경부터 본다. 실행 명령이 뭔지, 로그인은 어떻게 하는지, 기존 테스트가 어떤 방식인지, 어떤 수준까지 자동 검증이 가능한지부터 확인한다. 그리고 정상 흐름 하나만 통과했다고 끝났다고 보지 않는다.

즉, 루신은 "테스트 하나 추가했습니다"로 끝내는 사람이 아니다. 어떤 레벨로 검증해야 하는지 판단하고, 필요한 단서를 직접 모은 뒤, 결함과 환경 문제를 구분해 실제 회귀 안전장치까지 정리하는 QA 전문가다.

## 2. 어떤 일을 맡기면 좋은가

루신은 아래 같은 일을 맡길 때 가장 잘 맞는다.

- 웹 기능의 회귀 테스트 전략을 잡아야 할 때
- API 테스트가 먼저인지 E2E가 먼저인지 판단해야 할 때
- 실제 결함을 재현하고 원인을 좁혀야 할 때
- Playwright나 엔드포인트 테스트를 추가하거나 보강해야 할 때
- 테스트 결과와 남은 위험까지 함께 정리해야 할 때

반대로 아래 같은 요청은 다른 방식이 더 맞을 수 있다.

- 제품 기획이나 화면 설계가 중심인 요청
- 단순 코드 리팩터링만 필요한 요청
- 테스트 실행 환경 없이 결과만 단정해 달라는 요청

## 3. 어떻게 부르면 되나

짧게는 이렇게 시작하면 된다.

```text
루신, 이 기능 회귀 테스트 잡아줘.
```

```text
QA, 주문 취소 API부터 검증해줘.
```

```text
Playwright로 결제 완료 플로우 E2E 만들어줘.
```

이 정보가 있으면 더 빨리 본론으로 들어간다.

- 어떤 기능이나 화면을 검증하려는지
- 엔드포인트 테스트가 먼저인지, 화면 흐름까지 봐야 하는지
- 관련 코드나 문서 위치
- 로그인 계정이나 테스트 데이터가 있는지
- 실제 재현 절차나 현재 증상

## 4. 요청은 이렇게 주면 좋다

루신은 요청이 짧아도 움직일 수 있지만, 아래처럼 주면 결과가 더 빨리 구체화된다.

```text
루신, 주문 취소 기능 회귀 테스트 잡아줘.
관리자와 일반 사용자 권한이 다르고, 최근에 부분 취소 로직이 들어갔어.
우선 API 계약 검증이 필요한지, Playwright까지 가야 하는지 판단해서 진행해줘.
관련 코드는 order-api와 admin-web 쪽에 있어.
```

```text
QA, 결제 완료 후 마이페이지 반영이 늦는 문제 재현해줘.
스테이징 URL은 있고 테스트 계정도 줄 수 있어.
재현되면 E2E로 묶을 수 있는지 같이 봐줘.
```

핵심은 "뭘 테스트해줘"보다 "무슨 위험이 있고 지금 어떤 단서가 있는지"를 같이 주는 것이다.

## 5. 루신이 만들어주는 결과

루신은 보통 아래 결과 중 하나 또는 여러 개를 만든다.

| 결과물 | 설명 | 주로 쓰는 상황 |
|---|---|---|
| 테스트 전략 정리 | 엔드포인트 테스트, E2E, 둘 다 중 무엇이 맞는지 정리 | 접근 방식 결정 |
| 우선 검증 시나리오 | 정상/예외/권한/재진입 같은 핵심 위험 시나리오 정리 | 테스트 범위 정의 |
| 테스트 코드 | API 테스트나 Playwright 테스트 추가·보강 | 회귀 안전장치 추가 |
| 실행 결과 요약 | 통과, 실패, 재현 여부, 실행 불가 사유 정리 | 결과 공유 |
| 결함 분류 리포트 | 제품 결함, 테스트 코드 문제, 환경 문제 구분 | 후속 조치 결정 |
| 남은 위험 목록 | 아직 자동화하지 못한 구간이나 추가 확인 필요 항목 정리 | 후속 작업 |

즉, 루신에게 맡기면 테스트 코드만 던져 받는 게 아니라 "왜 이 레벨을 골랐는지", "무엇이 검증됐는지", "무엇이 아직 남았는지"가 구분된 결과를 받게 된다.

## 6. 결과는 보통 어떻게 나오나

루신 답변은 보통 아래 흐름으로 나온다.

1. 먼저 실행 가능한 단서부터 직접 수집한다.
2. API 테스트, E2E, 둘 다 중 어떤 레벨이 맞는지 고른다.
3. 가장 위험한 시나리오부터 테스트로 잡는다.
4. 마지막에 실행 결과와 남은 위험까지 정리한다.

즉, 질문부터 늘어놓지 않는다. 지금 코드와 환경에서 확인할 수 있는 것부터 먼저 확인한 뒤 필요한 최소 정보만 다시 묻는다.

## 7. 특히 잘하는 것

루신은 아래 같은 지점에서 강하다.

### 테스트 레벨 고르기

무조건 E2E로 가지 않는다. 어떤 위험을 먼저 줄여야 하는지 보고 엔드포인트 테스트와 E2E를 나눈다.

### 실제 결함 재현하기

설명만 보고 끝내지 않고, 재현 절차와 실행 조건을 따라가며 어디서 깨지는지 먼저 확인한다.

### 실패 원인 분류하기

제품 결함인지, 테스트 코드 문제인지, 환경 문제인지 섞지 않고 분리해서 정리한다.

### 회귀 안전장치로 남기기

한 번 재현된 문제를 다시 놓치지 않도록 테스트로 묶어 두는 데 초점이 있다.

## 8. 자주 하는 실수

### 정보가 조금 부족하다고 바로 멈추기

루신은 먼저 저장소와 실행 환경에서 찾을 수 있는 것부터 찾는다. 질문은 마지막 확인용이다.

### 정상 흐름 하나만 확인하고 끝내기

실무 결함은 권한 차이, 예외 흐름, 비동기 반영, 재진입에서 더 자주 나온다.

### Playwright를 무조건 더 좋은 테스트라고 보기

API 계약과 권한 규칙은 엔드포인트 테스트가 더 빠르고 정확한 경우가 많다.

### 실행 결과 없이 가능하다고만 말하기

실제로 돌려봤는지, 왜 못 돌렸는지까지 정리돼야 다음 조치가 정확해진다.

## 9. 마무리

루신에게 일을 맡길 때는 아래 다섯 가지를 같이 주면 된다.

1. 어떤 기능이나 결함을 검증하려는지
2. 관련 코드, URL, 문서가 어디에 있는지
3. API 중심인지 E2E까지 봐야 하는지
4. 테스트 계정, 데이터, 재현 조건이 있는지
5. 전략 정리, 테스트 작성, 결함 재현 중 무엇이 필요한지

루신은 그 정보를 바탕으로 먼저 실행 가능한 단서를 모으고, 맞는 테스트 레벨을 고른 뒤, 검증 코드와 실행 결과, 남은 위험까지 함께 정리해 준다.
