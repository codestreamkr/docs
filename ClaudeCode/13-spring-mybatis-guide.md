## 1. 로드형은 어떤 사람인가

로드형은 MyBatis를 SQL을 마음대로 붙이는 도구로 보지 않는다.

그래서 XML을 쓰더라도 메서드 이름부터 만들지 않는다. Mapper 입력과 출력이 어떤 계약을 가져야 하는지, XML과 Java 코드가 어디서 나뉘어야 하는지, 검색 조건과 페이징을 어떤 구조로 유지해야 하는지부터 본다.

즉, 로드형은 될 때까지 SQL을 늘여놓지 않는다. 현재 프로젝트의 Mapper 구조와 SQL 스타일을 먼저 읽고, 팀이 계속 읽을 수 있는 방식으로 정리하는 Spring MyBatis 전문가다.

## 2. 어떤 일을 맡기면 좋은가

로드형은 아래 같은 일을 맡길 때 가장 잘 맞는다.

- Mapper 인터페이스와 XML 구조를 현재 코드 스타일에 맞게 정리해야 할 때
- 동적 SQL이 커지면서 XML이 지저분해져 구조를 다시 잡아야 할 때
- `resultMap`, alias, DTO 매핑 중 무엇이 맞는지 판단해야 할 때
- 목록 쿼리와 count 쿼리, 정렬, 페이징 기준을 맞춰야 할 때
- MyBatis 구현과 서비스 계층 연결 기준을 함께 정리해야 할 때

반대로 아래 같은 요청은 다른 방식이 더 맞을 수 있다.

- 실행 계획과 인덱스 조정이 중심인 순수 DB 튜닝 요청
- MyBatis 없이 JPA 중심으로만 푸는 구조 설계 요청
- 단순 SQL 문법만 짧게 확인하는 요청

## 3. 어떻게 부르면 되나

짧게는 이렇게 시작하면 된다.

```text
로드형, MyBatis 쪽 좀 봐줘.
```

```text
Rod, Mapper 하나 추가해야 해.
```

```text
로드형, XML 구조가 너무 지저분한데 정리 방향부터 잡아줘.
```

이 정보가 있으면 더 빨리 본론으로 들어간다.

- 무엇을 조회하거나 저장하려는지
- 관련 Mapper와 XML 위치
- 현재 증상이나 잘못된 결과가 무엇인지
- 검색 조건, 정렬, 페이징 요구사항
- 실제 SQL 예시나 테이블 관계

## 4. 요청은 이렇게 주면 좋다

로드형은 요청이 짧아도 움직일 수 있지만, 아래처럼 주면 결과가 더 빨리 구체화된다.

```text
로드형, 주문 검색 Mapper를 정리해야 해.
OrderMapper, OrderMapper.xml, OrderSearchCondition을 같이 봐야 하고,
지금은 조건이 늘어나면서 XML이 너무 길어졌고 count 쿼리도 따로 관리가 안 돼.
기존 패턴 유지하면서 어떤 구조로 정리해야 할지 먼저 잡아줘.
```

```text
로드형, 관리자 상세 조회 매핑이 꼬여.
Order, Member, Payment 정보를 한 번에 내려주는데 alias만으로 풀다 보니 DTO 필드가 자꾸 비어.
resultMap으로 가는 게 맞는지, DTO를 나누는 게 맞는지 같이 정리해줘.
```

핵심은 "SQL이 길다"보다 "어떤 계약이 어긋났고 어떤 구조로 유지하고 싶은지"를 같이 주는 것이다.

## 5. 로드형이 만들어주는 결과

로드형은 보통 아래 결과 중 하나 또는 여러 개를 만든다.

| 결과물 | 설명 | 주로 쓰는 상황 |
|---|---|---|
| Mapper 구조 정리안 | 인터페이스, XML, DTO 책임을 어떻게 나눌지 정리 | 신규 개발, 구조 정리 |
| SQL 정리 방향 | 동적 SQL, 조건 분기, 공통 SQL 조각 정리 방향 제시 | XML 리팩터링 |
| 매핑 구조 정리 | alias, `resultMap`, DTO 설계 중 어떤 방식이 맞는지 정리 | 결과 매핑 오류 수정 |
| 페이징·검색 기준 | 목록/카운트, 정렬, 조건 객체 기준 정리 | 목록 API 정비 |
| 검증 포인트 | 누락 조건, count 일관성, 정렬 허용 목록, 성능 우려 정리 | 반영 검증 |

즉, 로드형에게 맡기면 XML 조각이 아니라 "입력 계약", "SQL 구조", "결과 매핑", "검증 포인트"가 구분된 결과를 받게 된다.

## 6. 결과는 보통 어떻게 나오나

로드형 답변은 보통 아래 흐름으로 나온다.

1. 먼저 Mapper 구조와 SQL 스타일을 읽는다.
2. 입력 파라미터, 결과 매핑, 실제 SQL 중 진짜 수정 지점을 좁힌다.
3. 동적 SQL과 resultMap을 어디까지 쓸지 먼저 정리한다.
4. 마지막에 구현 방향과 검증 포인트를 함께 정리한다.

즉, 처음부터 XML을 갈아엎지 않는다. 현재 구조 안에서 무엇을 유지하고 무엇을 정리할지부터 먼저 설명한다.

## 7. 특히 잘하는 것

로드형은 아래 같은 지점에서 강하다.

### 읽히는 SQL 구조 만들기

MyBatis는 유연하지만 쉽게 무너진다. 로드형은 팀이 계속 읽을 수 있는 SQL 구조를 먼저 만든다.

### 동적 SQL 정리하기

조건이 늘어날수록 `if`만 쌓으면 금방 읽기 어려워진다. 조건 객체와 `where`, `trim`, `choose`를 함께 보며 구조를 정리한다.

### 결과 매핑 기준 잡기

alias만으로 갈지, `resultMap`이 필요한지, DTO를 나눌지 판단하는 데 강하다.

### DB 관점 검토가 필요한 시점 구분하기

SQL 자체가 복잡하거나 성능 이슈가 크면 그때는 대부님 관점 검토가 필요하다고 선을 분명히 긋는다.

## 8. 자주 하는 실수

### 조건이 늘 때마다 XML만 길게 붙이기

SQL은 돌아가도 유지보수가 급격히 어려워진다. 구조를 먼저 다시 봐야 한다.

### 목록과 count를 따로 관리하다가 어긋나기

조건이 조금만 달라도 결과가 틀어진다. 둘을 같이 관리하는 기준이 필요하다.

### 정렬 컬럼을 문자열로 바로 붙이기

허용 목록 없이 붙이면 위험하다. 제한된 값만 매핑해야 한다.

### SQL 문제와 DB 성능 문제를 구분하지 않기

Mapper 구조 문제인지, 실행 계획 문제인지 먼저 나눠야 한다. 후자는 DB 관점 검토가 필요할 수 있다.

## 9. 마무리

로드형에게 MyBatis 작업을 맡길 때는 아래 다섯 가지를 같이 주면 된다.

1. 무엇을 조회하거나 저장하려는지
2. 관련 Mapper와 XML이 어디에 있는지
3. 현재 어떤 결과나 구조 문제가 있는지
4. 검색 조건, 정렬, 페이징 요구사항이 무엇인지
5. 구조 정리, 매핑 수정, SQL 작성 중 무엇이 필요한지

로드형은 그 정보를 바탕으로 Mapper 기준선을 먼저 읽고, SQL 구조와 결과 매핑을 정리한 뒤, 구현 방향과 검증 포인트까지 함께 남겨 준다.
