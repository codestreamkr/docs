## 1. 호페형은 무엇을 하는가

호페형은 외부 시스템 연동 구조를 분석하고 설계 문서로 정리하는 스킬이다.

- 결제, 인증, 메시징, 정산, 웹훅, 콜백, 벤더 API 연동 구조 분석
- 현재 구조 리포트 작성
- 목표 구조와 전환 순서 제안
- 개발 관점 정리와 문서 정리 작업을 나눠 동시에 진행
- 결과를 최종 문서 1개로 통합

핵심은 해결책부터 쓰지 않고, 현재 구조와 책임 경계를 먼저 정리하는 데 있다.

## 2. 무엇을 요청하면 좋은가

```text
호페형, 결제 연동 구조 분석해줘.
```

```text
호페형, KCP에서 토스로 이관하는 구조 설계해줘.
```

```text
external-architect, 웹훅 재처리까지 포함해서 문서로 정리해줘.
```

- 현재 연동 구조를 먼저 정리해야 할 때
- 특정 벤더 기준으로 설계를 다시 잡아야 할 때
- 벤더 이관이나 공존 기간 운영 기준이 필요할 때
- 웹훅, 콜백, 상태 전이, 멱등 처리 기준을 문서로 확정해야 할 때

- 분석인지, 설계인지, 둘 다인지
- 현재 벤더와 대상 벤더
- 관련 코드나 문서 위치
- 최종 산출물이 리포트인지 설계 문서인지

## 3. 무엇을 하지 않는가

- 첫 응답에서 바로 분석이나 설계 문장을 쓰지 않는다
- 체크리스트 전에 저장소, 문서, 공식 문서를 읽지 않는다
- 현재 구조 리포트 없이 목표 구조를 먼저 제안하지 않는다
- 공식 문서로 확인되지 않은 상태값이나 필드를 단정하지 않는다
- 개발 상세 정리와 문서 정리 역할을 한 사람이 모두 대신하지 않는다
- 역할 분담 없이 최종 문서를 완료 처리하지 않는다

## 4. 어떻게 진행하는가

1. 요청을 `분석`, `설계`, `분석+설계`로 분류한다.
2. 첫 응답으로 체크리스트만 작성한다.
3. 주 대상 벤더를 정하고 해당 벤더의 안내 문서를 확인한다.
4. 현재 구조 리포트를 작성한다.
5. 리포트가 끝난 뒤 설계 제안을 작성한다.
6. 개발 상세 정리와 문서 정리 작업을 같은 시점에 나눠 진행한다.
7. 두 결과를 검수한 뒤 최종 문서 1개로 통합한다.

## 5. 첫 응답은 왜 체크리스트만 쓰는가

첫 응답은 체크리스트만 쓴다. 설명이나 질문은 붙이지 않는다.

- 현재 연동 진입점 확인
- 벤더 호출 모듈 확인
- 응답 및 콜백 수신 경로 확인
- 상태 저장 위치 확인
- 트랜잭션 경계 확인
- 멱등과 재처리 구조 확인
- 운영 로그와 모니터링 확인
- 공식 벤더 문서와 비교
- 리포트 작성
- 제안 작성
- 역할 분담 후 병렬 진행
- 최종 문서 통합

## 6. 리포트 단계에서 무엇을 확인하는가

리포트 단계에서는 현재 구조만 정리한다.

- 외부 호출 시작 지점
- 벤더 클라이언트, 어댑터, 서비스 경계
- 리턴 URL, 웹훅, 배치, 재처리 진입점
- 결제 또는 연동 상태 저장 위치
- 외부 호출 전후 트랜잭션 경계
- 멱등 키, 중복 방지, 재시도 정책
- 로그, 추적 ID, 마스킹, 알림 구조
- 테스트 범위와 모킹 방식

1. 요청 분류
2. 현재 구조 요약
3. 벤더 종속 지점
4. 상태 저장과 전이 구조
5. 실패 경로와 운영 위험
6. 확인 필요

## 7. 벤더 문서는 어떻게 보는가

체크리스트 뒤에는 주 벤더를 정하고 관련 문서를 확인한다.

- 토스페이먼츠
- NHN KCP
- 카카오페이
- 네이버페이

벤더가 불분명하면 현재 구조를 보고 주 벤더 하나를 정한다. 공식 문서로 확정되지 않은 내용은 `확인 필요`로 남긴다.

## 8. 제안 단계에서는 무엇을 정리하는가

제안 단계는 리포트 뒤에 진행한다.

- 목표 구조와 책임 경계
- 벤더 원본 상태와 내부 도메인 상태를 분리하는 방식
- 승인, 취소, 조회, 웹훅, 콜백 처리 원칙
- 멱등, 재처리, 중복 방지 기준
- 공존 기간 라우팅 기준
- 전환 순서와 롤백 기준
- 운영 체크포인트

1. 목표 구조
2. 책임 분리
3. 상태 전이 원칙
4. 오류 처리와 재처리 원칙
5. 전환 순서
6. 확인 필요

## 9. 후속 작업은 어떻게 나누는가

후속 작업은 같은 시점에 나눠 진행한다.

- 개발 상세 정리: 개발 가이드, 저장 모델, API 계약, 상태 전이, 멱등 처리, 구현 순서, 테스트 포인트 구체화
- 문서 정리: 최종 문서 보완, 구조 정리, 누락 보완, 표현 정리, 중복 제거, 이력관리 형식 정리

메인 진행자는 둘을 대신하지 않고, 결과를 검수해 최종본으로 묶는다.

## 10. 최종 문서는 어떻게 닫는가

1. 개발 상세 정리 결과를 반영한다.
2. 문서 정리 결과를 반영한다.
3. 리포트, 제안, 실무 구체화를 문서 1개로 통합한다.
4. 공식 문서로 확정되지 않은 내용은 `확인 필요`로 남긴다.
5. 문서 끝에 `## 이력관리`를 둔다.

## 11. 이런 요청에 특히 잘 맞는다

- 외부 결제 벤더 이관 설계가 필요한 경우
- 웹훅과 내부 상태를 어떻게 분리할지 정리해야 하는 경우
- 승인, 취소, 조회, 배치, 재처리 흐름을 한 문서로 묶어야 하는 경우
- 운영 기준과 롤백 기준까지 포함한 전환 설계가 필요한 경우

반대로 바로 구현 코드만 원하거나, 역할을 나누지 않고 한 번의 답변으로 끝낼 작업이라면 호페형보다 다른 스킬이 더 맞을 수 있다.
