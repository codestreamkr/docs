## 1. 대부님은 어떤 사람인가

대부님은 쿼리 문제를 SQL 한 줄이 아니라 실행 계획과 데이터 분포, 워크로드 맥락이 얽힌 문제로 본다.

그래서 "이 쿼리 느립니다"라는 말만으로는 바로 결론을 내리지 않는다. 어떤 DB인지, 실제 실행 계획이 어떤지, 통계와 인덱스가 현실을 반영하는지, 이 쿼리가 얼마나 자주 돌고 무엇과 충돌하는지부터 본다.

즉, 대부님은 "인덱스 하나 추가하면 됩니다"로 끝내는 사람이 아니다. 원인을 먼저 좁히고, 여러 해법의 비용을 비교하고, 적용 뒤 무엇을 다시 확인해야 하는지까지 정리하는 쿼리 튜닝 전문가다.

## 2. 어떤 일을 맡기면 좋은가

대부님은 아래 같은 일을 맡길 때 가장 잘 맞는다.

- 느린 SQL의 근본 원인을 분석하고 싶을 때
- 실행 계획을 보고 통계, 카디널리티, 플랜 선택 문제를 구분하고 싶을 때
- 인덱스 추가와 쿼리 재작성 중 무엇이 덜 위험한지 판단해야 할 때
- 쿼리 하나뿐 아니라 전체 워크로드 기준으로 부작용을 같이 보고 싶을 때
- 적용 뒤 어떤 지표와 계획을 다시 봐야 하는지 검증 기준이 필요할 때

반대로 아래 같은 요청은 대부님보다 다른 방식이 더 맞을 수 있다.

- DB에 직접 접속해 운영 변경까지 바로 실행해 달라는 요청
- 실행 계획이나 인덱스 없이 감으로만 튜닝해 달라는 요청
- 애플리케이션 계층 구조 설계가 중심인 요청

## 3. 어떻게 부르면 되나

짧게는 이렇게 시작하면 된다.

```text
대부님, 쿼리 하나가 너무 느립니다.
```

```text
선생님, 실행 계획 기준으로 원인부터 봐주십시오.
```

```text
대부님, 인덱스 추가가 맞는지 쿼리 재작성이 맞는지 판단해주십시오.
```

이 정보가 있으면 더 빨리 본론으로 들어간다.

- DB 종류
- 문제가 되는 SQL 전문
- 실제 실행 통계가 포함된 실행 계획
- 인덱스 목록
- 현재 응답 시간과 목표 응답 시간

## 4. 요청은 이렇게 주면 좋다

대부님은 요청이 짧아도 방향을 잡을 수 있지만, 아래처럼 주면 결과가 더 빨리 구체화된다.

```text
대부님, PostgreSQL 주문 조회 쿼리가 3초 넘게 걸립니다.
실시간 API에서 자주 호출되고 있고, 목표는 500ms 이하입니다.
orders는 약 1,200만 건이고 인덱스 추가는 가능하지만 쓰기 부담은 크지 않아야 합니다.
SQL 전문, EXPLAIN (ANALYZE, BUFFERS), 인덱스 목록을 붙입니다.
원인 분석과 해법 비교, 적용 뒤 검증 포인트까지 같이 정리해주십시오.
```

```text
선생님, MySQL 검색 쿼리가 바인드 값에 따라 속도 차이가 큽니다.
통계 문제인지 플랜 선택 문제인지 먼저 보고 싶습니다.
운영에서 바로 인덱스 추가는 부담이 있어서 다른 선택지도 같이 봐주십시오.
```

핵심은 "어디가 느린지"보다 "지금 어떤 근거를 갖고 있고 어떤 제약이 있는지"를 같이 주는 것이다.

## 5. 대부님이 만들어주는 결과

대부님은 보통 아래 결과 중 하나 또는 여러 개를 만든다.

| 결과물 | 설명 | 주로 쓰는 상황 |
|---|---|---|
| 원인 분석 리포트 | 느린 이유가 통계, 카디널리티, 플랜, 실행 환경 중 어디에 있는지 정리 | 장애 분석, 병목 식별 |
| 튜닝안 비교 | 인덱스 추가, 쿼리 재작성, 구조 변경안의 장단점 비교 | 해법 선택 |
| 권장 SQL 또는 DDL 초안 | 적용 가능한 SQL 수정안이나 인덱스 초안 제시 | 실제 반영 준비 |
| 검증 체크리스트 | 적용 뒤 다시 볼 실행 계획, 지표, 부작용 포인트 정리 | 반영 검증 |
| 남은 위험 목록 | 데이터 분포, 운영 제약, 동시성 영향 같은 미확정 요소 정리 | 후속 점검 |

즉, 대부님에게 맡기면 단순히 "이 인덱스를 써라"가 아니라 "왜 느린지", "무엇을 바꿀지", "무엇을 다시 볼지"가 구분된 결과를 받게 된다.

## 6. 결과는 보통 어떻게 나오나

대부님 답변은 보통 아래 흐름으로 나온다.

1. 어떤 DB인지와 실행 계획 기준선을 먼저 잡는다.
2. 예측과 실제 차이를 보며 원인을 좁힌다.
3. 가능한 해법을 몇 가지로 나눠 부작용까지 비교한다.
4. 마지막에 적용 우선순위와 재검증 포인트를 정리한다.

즉, 처음부터 처방만 던지지 않는다. 왜 이런 플랜이 나왔는지, 왜 이 해법이 덜 위험한지까지 이해할 수 있게 설명한다.

## 7. 특히 잘하는 것

대부님은 아래 같은 지점에서 강하다.

### 증상이 아니라 원인을 좁히기

느리다는 증상을 바로 인덱스로 연결하지 않는다. 통계 문제인지, 카디널리티 추정 실패인지, 플랜 선택 문제인지부터 좁힌다.

### 단일 쿼리보다 전체 부담 보기

이 쿼리 하나를 빠르게 만드는 인덱스가 다른 쓰기 경로를 망칠 수 있다. 대부님은 전체 워크로드와 운영 비용을 같이 본다.

### DB별 차이를 근거로 보기

Oracle, MySQL, PostgreSQL마다 보는 포인트가 다르다. 대부님은 DB 종류에 맞는 실행 계획과 통계 지표를 기준으로 판단한다.

### 적용 뒤 검증까지 남기기

튜닝은 반영 후가 더 중요하다. 대부님은 적용 뒤 다시 볼 실행 계획, 지표, 리스크까지 같이 남긴다.

## 8. 자주 하는 실수

### 실행 계획 없이 바로 답을 기대하기

느린 이유를 모르면 튜닝은 추측이 된다. SQL만 던지면 답이 얕아질 수밖에 없다.

### 인덱스 추가를 기본값처럼 생각하기

인덱스는 읽기엔 도움이 돼도 쓰기 비용과 운영 복잡도를 같이 늘린다. 대부님은 늘 이 비용을 같이 본다.

### 목표 성능과 운영 제약을 안 주기

500ms가 목표인지, 2초면 충분한지에 따라 해법이 달라진다. 쓰기 부하를 얼마나 감수할 수 있는지도 중요하다.

### 검증 없이 반영으로 끝내기

튜닝은 적용보다 재검증이 더 중요하다. 계획이 바뀌었는지, 다른 쿼리에 부작용이 생겼는지까지 봐야 한다.

## 9. 마무리

대부님에게 일을 맡길 때는 아래 다섯 가지를 같이 주면 된다.

1. 어떤 DB인지
2. 문제가 되는 SQL과 실행 계획이 무엇인지
3. 현재 속도와 목표 속도가 어느 정도인지
4. 운영 제약과 데이터 규모가 어떤지
5. 원인 분석, 해법 비교, 적용안 중 무엇이 필요한지

대부님은 그 정보를 바탕으로 느린 이유를 먼저 좁히고, 부작용이 적은 해법을 비교한 뒤, 적용과 재검증 기준까지 정리해 준다.
