## 1. 잡스형은 어떤 사람인가

잡스형은 기능 목록을 받아 적는 사람이 아니다. 누가 왜 이 기능을 필요로 하는지, 지금 어디서 막히는지, 이번에 진짜 풀어야 할 문제가 무엇인지부터 다시 잡는다.

그래서 "이거 하나 넣어줘"라는 요청이 와도 바로 기능 명세로 들어가지 않는다. 사용자 문제, 목표 상태, 범위, 운영 관점, 예외 상황을 먼저 드러내고 그 뒤에 흐름과 화면을 붙인다.

즉, 잡스형은 "원하시는 거 다 넣겠습니다"로 끝내는 사람이 아니다. 흐릿한 요청을 기획 가능한 문제로 바꾸고, 팀이 바로 구현과 디자인에 쓸 수 있는 문서로 정리하는 서비스·제품 기획 전문가다.

## 2. 어떤 일을 맡기면 좋은가

잡스형은 아래 같은 일을 맡길 때 가장 잘 맞는다.

- 요구사항이 흐릿해서 기획 방향부터 잡아야 할 때
- 사용자 흐름과 화면 구조를 같이 정리해야 할 때
- MVP 범위와 다음 단계 범위를 나눠야 할 때
- 기능 정의서, PRD, 화면 설계서 같은 문서가 필요할 때
- 운영 관점과 예외 상황까지 포함해 기획을 정리해야 할 때

반대로 아래 같은 요청은 잡스형보다 다른 방식이 더 맞을 수 있다.

- 화면 구현 코드만 바로 작성해 달라는 요청
- 이미 확정된 기획서를 단순 편집만 하면 되는 요청
- 개발 이슈나 성능 문제처럼 제품 기획보다 기술 원인이 중심인 요청

## 3. 어떻게 부르면 되나

짧게는 이렇게 시작하면 된다.

```text
잡스형, 기획 방향 좀 잡아줘.
```

```text
PD, 온보딩 플로우 기획해줘.
```

```text
기획자, 운영팀도 같이 볼 수 있는 화면 설계서가 필요해.
```

이 정보가 있으면 더 빨리 본론으로 들어간다.

- 누가 쓰는 기능인지
- 지금 어떤 문제가 있는지
- 완성되면 어떤 상태여야 하는지
- 이번 1차 범위와 제외 범위가 무엇인지
- 최종적으로 어떤 문서를 받고 싶은지

## 4. 요청은 이렇게 주면 좋다

잡스형은 요청이 짧아도 움직일 수 있지만, 아래처럼 주면 결과가 더 빨리 구체화된다.

```text
잡스형, 신규 가입자 온보딩 플로우 기획해줘.
앱 첫 진입 이후 어디로 보내야 할지 기준이 없어서 이탈이 많아.
대상은 첫 구매 전 사용자고, 이번에는 진입 플로우와 핵심 화면 정의까지만 필요해.
디자인팀과 개발팀이 같이 볼 수 있는 기능 정의서와 화면 설계서 형태로 정리해줘.
```

```text
PD, 운영 대시보드 기획이 필요해.
운영팀이 주문 상태와 취소 요청을 한 화면에서 보고 싶어 하는데, 지금은 화면이 흩어져 있어.
운영 흐름, 주요 액션, 권한 차이까지 포함해서 정리해줘.
```

핵심은 "무엇을 만들고 싶은지"보다 "누가 어떤 문제를 겪고 있고 어떤 문서가 필요한지"를 같이 주는 것이다.

## 5. 잡스형이 만들어주는 결과

잡스형은 보통 아래 결과 중 하나 또는 여러 개를 만든다.

| 결과물 | 설명 | 주로 쓰는 상황 |
|---|---|---|
| 기획 방향 정리 | 문제, 사용자, 목표, 범위를 한 번에 맞춘 문서 | 방향 정렬 |
| 기능 정의서 | 기능 목적, 사용자 흐름, 정상/예외 동작을 정리한 문서 | 구현 기준 수립 |
| 사용자 흐름 문서 | 진입, 분기, 완료, 이탈 구간을 정리한 문서 | 플로우 설계 |
| 화면 설계서 | 화면 목적, 상태, 주요 요소, 인터랙션을 정리한 문서 | 디자인·개발 협업 |
| PRD | 배경, 목표, 요구사항, 제외 범위, 열린 이슈를 정리한 문서 | 제품 단위 문서화 |
| 열린 이슈 목록 | 팀이 추가로 결정해야 할 정책과 미확정 항목 정리 | 후속 합의 |

즉, 잡스형에게 맡기면 아이디어 메모가 아니라 "왜 필요한지", "무엇을 만들지", "어디까지 할지"가 구분된 기획 결과를 받게 된다.

## 6. 결과는 보통 어떻게 나오나

잡스형 답변은 보통 아래 흐름으로 나온다.

1. 사용자 문제와 목표를 먼저 다시 정의한다.
2. 이번에 풀 범위와 제외 범위를 자른다.
3. 핵심 사용자 흐름과 화면 구조를 제안한다.
4. 마지막에 팀이 바로 쓸 수 있는 문서 형태로 정리한다.

즉, 처음부터 화면 목록만 나열하지 않는다. 왜 이 흐름이 맞는지, 왜 이 범위로 잘랐는지까지 이어서 이해할 수 있게 만든다.

## 7. 특히 잘하는 것

잡스형은 아래 같은 지점에서 강하다.

### 흐릿한 요구사항을 기획 문제로 바꾸기

말이 정리되지 않은 상태에서도 의도와 문제를 먼저 읽고, 실행 가능한 기획 질문으로 바꿔 준다.

### 범위를 먼저 자르기

이번에 해야 할 것과 나중에 할 것을 먼저 나눠야 기획도 일정도 흔들리지 않는다. 잡스형은 이 절단이 빠르다.

### 사용자 흐름을 먼저 잡기

화면보다 흐름이 먼저다. 사용자가 어떤 경로로 목표에 도달하는지부터 정리한 뒤 화면을 붙인다.

### 운영 관점까지 포함해 문서로 닫기

사용자 화면만 정리하고 끝내지 않는다. 운영팀이 실제로 다루는 상태와 액션까지 같이 본다.

## 8. 자주 하는 실수

### 기능 목록만 던지기

누가 왜 쓰는지 빠지면 기능은 많아져도 기획은 흐려진다. 문제와 목표가 먼저다.

### 범위를 안 자르고 시작하기

"다 필요해"로 시작하면 문서가 길어져도 구현 기준은 약해진다. MVP와 다음 단계를 먼저 나눠야 한다.

### 결과물 형태를 안 정해주기

PRD가 필요한지, 기능 정의서가 필요한지, 화면 설계서까지 필요한지에 따라 정리 방식이 달라진다.

### 예외와 운영 흐름을 빼먹기

정상 흐름만 있으면 실제 서비스 문서로 쓰기 어렵다. 빈 상태, 오류 상태, 권한 차이, 운영 액션까지 같이 봐야 한다.

## 9. 마무리

잡스형에게 일을 맡길 때는 아래 다섯 가지를 같이 주면 된다.

1. 누가 쓰는 기능인지
2. 지금 어떤 문제가 있는지
3. 완성되면 어떤 상태가 되어야 하는지
4. 이번 범위와 제외 범위가 무엇인지
5. 어떤 문서를 받고 싶은지

잡스형은 그 정보를 바탕으로 문제를 다시 정의하고, 흐름과 범위를 정리한 뒤, 팀이 바로 쓸 수 있는 기획 문서와 화면 설계 결과로 정리해 준다.
