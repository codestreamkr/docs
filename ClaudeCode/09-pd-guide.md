# 잡스형 첫번째 특집 - 서비스·제품 기획 전문가

요구사항 한 줄에서 기획서, 화면 설계, Figma 산출물까지 end-to-end로 밀어붙이는 잡스형 페르소나와 활용법을 정리한다.

## 1. 잡스형은 어떤 사람인가

잡스형은 요구사항을 받아서 정리하는 사람이 아니다.  
아직 아무도 묻지 않은 질문을 먼저 찾아내고, 그 답을 화면과 문서로 만들어내는 사람이다.

말투는 짧고 단호하다.  
"이건 틀렸어", "이렇게 가야 해", "왜 이게 필요해?" — 돌려 말하지 않는다.  
모호한 요구사항이 들어오면 그냥 받지 않는다. 살을 붙여서 역제안으로 돌아온다.

기획서가 먼저다.  
방향이 글로 정리되지 않은 상태에서 Figma부터 여는 일은 없다.  
기획이 안 잡힌 상태에서 화면을 그리면 나중에 전부 다시 한다는 걸 알기 때문이다.

즉, 잡스형은 "원하시는 거 다 넣어드릴게요"로 끝나는 사람이 아니다.  
왜 그게 필요한지부터 묻고, 불필요한 건 쳐내는 기획 전문가다.

## 2. 제일 쉬운 시작

처음에는 아주 짧게 불러도 된다.

```text
잡스형, 도와줘
```

또는

```text
PD, 기획 좀 봐줘
```

그러면 잡스형은 대개 이런 식으로 받아준다.

```text
뭘 만들려는 거야? 이 서비스가 누구를 위한 건지, 어떤 문제를 푸는 건지부터 말해봐.
기능 목록 먼저 들고 오는 건 의미 없어. 방향이 먼저야.
```

이 정도만으로도 잡스형 페르소나는 충분히 살아난다.  
처음부터 완벽하게 정리해서 들어갈 필요는 없다.  
다만 더 빠르게 산출물을 받고 싶다면, 아래처럼 조금만 더 얹어주면 된다.

## 3. 처음부터 잘 묻는 방법

처음엔 짧게 시작해도 되지만, 아래 정보를 같이 주면 진단 속도와 산출물 품질이 눈에 띄게 좋아진다.

### 꼭 있으면 좋은 것

- 대상 사용자 — 누가 쓰는 서비스인가
- 해결할 문제 — 사용자가 지금 어떤 불편을 겪고 있나
- 목표 상태 — 완성됐을 때 어떤 상태여야 하나
- 1차 범위 — 이번에 반드시 포함해야 할 것

### 있으면 더 좋아지는 것

- 비즈니스 목표 — 매출, 리텐션, 신규 유입 중 어느 것이 우선인가
- 제외 범위 — 이번에는 하지 않을 것
- 기술 제약 — 연동해야 할 시스템, 못 쓰는 스택
- 기한 — 기획서 또는 화면 완료 기한

잡스형은 이 정보를 바탕으로 "기능 요청"이 아니라 "사용자 문제"를 본다.  
그래서 기능 목록보다 누가, 왜 쓰는지가 훨씬 중요하다.

## 4. 질문은 이렇게 점점 좋아진다

같은 요청이라도 어떻게 던지느냐에 따라 산출물의 깊이가 달라진다.  
처음부터 완벽하게 준비할 필요는 없지만, 아래 흐름으로 갈수록 잡스형이 더 빠르게 본론으로 들어간다.

### 4.1 조금 더 좋은 호출

```text
잡스형, 온보딩 플로우 기획해야 해.
신규 가입자가 앱 처음 켰을 때 어디로 가야 할지 정리가 안 돼 있어.
```

이 정도만 줘도 잡스형은 온보딩의 목적과 이탈 구간을 기준으로 다음에 무엇이 필요한지 역제안으로 돌아온다.

비슷한 방식으로는 아래처럼 시작해도 된다.

```text
PD, 결제 플로우 기획 좀 봐줘.
지금 결제 완료 후 어디로 가야 하는지 정해진 게 없어.
```

```text
기획자, 대시보드 만들어야 해.
운영팀이 주문 현황 보려면 어떤 화면이 필요한지 모르겠어.
```

### 4.2 실전형 호출

```text
잡스형, 푸시 알림 기능 기획해야 해.
구매 완료 후 7일이 지난 사용자한테 재구매 유도 알림을 보내고 싶어.
대상은 30~40대 직장인이고, 지금 앱 재방문율이 20%인데 35%까지 올리는 게 목표야.
이번엔 알림 발송과 설정 화면만 포함하고, 개인화 추천은 다음 단계야.
기획서 + 화면 설계서까지 필요해.
```

여기까지 주면 잡스형은 바로 본론으로 들어갈 수 있다.  
이 단계부터는 "어떤 기능이 필요해요?"가 아니라 "왜 이 플로우가 맞는가"를 본격적으로 파고든다.

다른 기획 요청에서는 아래처럼 바꿔 써도 된다.

```text
잡스형, 회원가입 플로우 리뉴얼해야 해.
지금 가입 완료율이 너무 낮고, 이메일 인증 단계에서 이탈이 많아.
소셜 로그인은 이미 있고, 이번엔 이메일 가입 플로우만 개선하면 돼.
기획서부터 뽑아줘.
```

```text
PD, 관리자 어드민 화면 기획 잡아줘.
고객센터 팀이 환불 요청 처리하는 화면이 없어서 지금 개발팀한테 직접 요청하고 있어.
이번에 환불 조회, 승인/반려, 처리 내역만 먼저 만들어.
```

## 5. 잡스형은 무엇부터 보나

잡스형은 보통 아래 순서로 본다.

1. 사용자가 누구인지 확인한다
2. 해결할 문제가 실제로 존재하는지 검증한다
3. 범위를 먼저 잘라낸다 — 할 것과 안 할 것을 분리한다
4. 사용자 플로우를 화면보다 먼저 정의한다
5. 기획서 1차 초안을 완성한다
6. 팀 결정이 필요한 포인트를 명확히 표시한다
7. 엣지케이스와 운영 관점을 상세화한다
8. 외부 툴(Figma 등)은 기획서가 나온 뒤 원할 때만 진행한다

즉, 답만 던지고 끝내지 않는다.  
왜 이 플로우인지, 어떤 상태를 빠뜨렸는지, 운영 관점은 잡혔는지까지 함께 본다.

## 6. 산출물 유형

잡스형이 요청에 따라 만들어주는 산출물 목록이다.

| 산출물 | 설명 | 주요 용도 |
|---|---|---|
| 서비스 정의서 | 누구를 위해, 무슨 문제를, 어떻게 해결하는지 한 장으로 정리 | 팀 방향 정렬 |
| 기능 정의서 | 개발·디자인 팀이 실제로 쓸 수 있는 기능 명세 | 구현 기준 |
| 사용자 플로우 | 진입부터 완료까지의 경로, 분기, 엣지케이스 | 화면 설계 전 선행 작업 |
| 화면 설계서 | 각 화면의 구성 요소, 동작, 상태 정의 | 디자인·QA 기준 |
| PRD | 제품 단위 요구사항 문서, 기능 정의서의 집합 | 릴리즈 기준 |
| Figma 와이어프레임 | 기획서 기반 화면 시각화 | 시각적 검토, 프로토타이핑 |

사용자는 원하는 산출물만 지정하면 된다.  
지정하지 않으면 잡스형이 요청 맥락에 맞게 판단해서 시작한다.

## 7. 역제안이 작동하는 방식

잡스형의 핵심은 역제안이다.  
완성된 요청을 기다리지 않고, 흘리듯 던진 말에서 먼저 방향을 잡아 돌아온다.

### 역제안 예시

> 사용자: "우리 앱 온보딩이 좀 길어"  
> 잡스형: "온보딩이 길다는 게 문제가 아니야. 왜 긴지가 문제다. 단계가 많은 건지, 각 단계에서 입력이 많은 건지 — 이 두 개는 해법이 완전히 달라. 현재 몇 단계야? 어느 단계에서 이탈이 제일 많아?"

> 사용자: "대시보드 만들어야 해"  
> 잡스형: "누가 보는 거야? 운영팀이 보는 관리 도구인지, 사용자가 직접 보는 셀프서비스인지에 따라 완전히 다른 물건이 나온다. 그리고 대시보드에서 뭔가를 해야 해, 아니면 그냥 보기만 해?"

> 사용자: "결제 플로우 기획해줘"  
> 잡스형: "결제 자체보다 주변이 더 복잡해. 결제 성공 후 다음 화면은 뭔지, 실패했을 때 사용자한테 뭘 보여주는지가 먼저다. 운영 쪽도 처음부터 같이 잡아야 해. 환불은 즉시야, 아니면 심사 후야?"

방향이 합의되면 잡스형은 질문에서 멈추지 않고 바로 산출물로 붙는다.

## 8. 자주 하는 실수

### 기능 목록부터 던지기

잡스형은 기능 목록을 받으면 바로 작성하지 않는다.  
누가 쓰는 기능인지, 왜 필요한지 먼저 물어온다.  
기능이 먼저가 아니라 사용자 문제가 먼저다.

### 범위를 정하지 않고 시작하기

"다 만들어줘"는 기획의 적이다.  
1차에 할 것과 나중에 할 것을 나눠야 기획서도 제대로 나온다.  
범위가 불분명하면 잡스형이 먼저 잘라낸다.

### Figma부터 요청하기

기획이 안 잡힌 상태에서 화면부터 그리면 나중에 전부 다시 한다.  
잡스형은 기획서가 나오지 않은 상태에서 Figma를 열지 않는다.  
화면 설계 요청은 플로우 합의 이후에 한다.

### 산출물 없이 구두로만 정리하기

논의 결과는 반드시 파일로 남아야 한다.  
"나중에 정리할게요"로 흘리면 잡스형이 바로 끊고 문서부터 만든다.

## 9. 이런 답변을 기대하면 된다

잡스형 답변은 보통 아래 흐름으로 정리된다.

- 요청에서 읽은 의도 정리
- 빠진 것 또는 재고가 필요한 것 지적
- 방향 역제안
- 합의 후 즉시 기획서 초안 또는 화면 목록 작성
- 팀 결정이 필요한 열린 이슈 명시
- 다음 단계 제시 (상세화, Figma, PRD 등)

즉, 답만 주는 것이 아니라 왜 그 방향인지와 다음에 무엇을 결정해야 하는지까지 함께 준다.

## 10. 마무리

핵심은 단순하다.  
기획 요청을 들고 갈 때 기능 목록부터 채우지 말고, 누가 쓰는지·왜 필요한지·이번에 무엇만 할 건지를 먼저 정리해 가는 것이다.  
잡스형은 나머지를 역제안으로 채워준다.
