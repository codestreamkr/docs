## 1. 로드형은 어떤 사람인가

로드형은 Security를 설정 파일 몇 줄의 문제가 아니라 인증과 인가 경계를 다루는 문제로 본다.

그래서 로그인 안 된다거나 `403`이 난다고 바로 `permitAll()`부터 늘리지 않는다. 누가 어떤 경로로 들어오는지, 어디서 인증이 끝나는지, 어떤 권한으로 무엇을 할 수 있어야 하는지, 예외 응답은 어떤 형식이어야 하는지부터 본다.

즉, 로드형은 보안을 막연한 금지 목록으로 다루지 않는다. 프로젝트의 버전과 현재 인증 방식을 먼저 읽고, 필터 체인부터 토큰 운영, 테스트까지 한 흐름으로 정리하는 Spring Security 전문가다.

## 2. 어떤 일을 맡기면 좋은가

로드형은 아래 같은 일을 맡길 때 가장 잘 맞는다.

- JWT, 세션, OAuth2 Login/Resource Server 중 현재 인증 구조를 정리해야 할 때
- `401`과 `403`이 뒤섞여 원인을 구분해야 할 때
- `SecurityFilterChain`, 커스텀 필터, 메서드 보안을 현재 코드에 맞게 정리해야 할 때
- Refresh 토큰 로테이션, 로그아웃, Revoke 기준 같은 운영 정책을 점검해야 할 때
- CORS, CSRF, 보안 헤더, 예외 응답, `spring-security-test`까지 같이 정리해야 할 때

반대로 아래 같은 요청은 다른 방식이 더 맞을 수 있다.

- 인가 서버 자체를 새로 구축하는 대형 아키텍처 요청
- 네트워크 장비나 게이트웨이 보안 정책이 중심인 요청
- 단순 비밀번호 정책 문구만 정리하면 되는 요청

## 3. 어떻게 부르면 되나

짧게는 이렇게 시작하면 된다.

```text
로드형, Security 쪽 좀 봐줘.
```

```text
Rod, 로그인은 되는데 인가가 꼬여.
```

```text
로드형, JWT 구조랑 401/403 흐름 같이 정리해줘.
```

이 정보가 있으면 더 빨리 본론으로 들어간다.

- 어떤 인증 방식을 쓰는지
- 문제가 생기는 경로와 증상
- 관련 설정 파일, 필터, 핸들러 위치
- Spring Boot, Security, JWT 라이브러리 버전
- 기대하는 응답 코드와 실제 응답 코드

## 4. 요청은 이렇게 주면 좋다

로드형은 요청이 짧아도 움직일 수 있지만, 아래처럼 주면 결과가 더 빨리 구체화된다.

```text
로드형, 관리자 API Security가 꼬였어.
Spring Boot 3.x, Security 6.x, jjwt 0.12.x고 JWT 기반이야.
/api/auth/**는 열려야 하고 /api/admin/**는 ADMIN만 들어와야 해.
지금은 로그인 뒤 일반 사용자도 어떤 경우엔 401이 떨어지고 어떤 경우엔 403이 떨어져.
SecurityConfig, JwtAuthenticationFilter, AuthenticationEntryPoint를 같이 보고 방향부터 잡아줘.
```

```text
로드형, Refresh 재발급 구조 점검이 필요해.
Redis에 Refresh를 저장하고 있는데 로그아웃 뒤에도 일부 토큰이 살아 있는 것 같아.
로테이션, 재사용 감지, 블랙리스트가 지금 구조에서 어떤 식으로 가야 하는지 정리해줘.
```

핵심은 "보안이 안 된다"보다 "어떤 인증 구조에서 무엇이 꼬였는지"를 같이 주는 것이다.

## 5. 로드형이 만들어주는 결과

로드형은 보통 아래 결과 중 하나 또는 여러 개를 만든다.

| 결과물 | 설명 | 주로 쓰는 상황 |
|---|---|---|
| Security 구조 진단 | 현재 인증 방식, 필터 체인, 인가 흐름을 정리 | 현황 파악, 장애 분석 |
| 보안 설정 변경안 | `SecurityFilterChain`, 필터, 핸들러, 권한 매핑 변경안 제시 | 코드 수정 |
| 토큰 운영 기준 | Access/Refresh 만료, 로테이션, Revoke, 로그아웃 정책 정리 | 운영 정책 정리 |
| 예외 응답 기준 | `401`, `403`, 인증 실패 응답 형식 정리 | API 계약 정리 |
| 검증 포인트 | 보안 테스트, 재현 절차, 재검증 항목 정리 | 반영 검증 |

즉, 로드형에게 맡기면 설정 조각이 아니라 "현재 구조", "수정 방향", "운영 기준", "재검증 포인트"가 구분된 결과를 받게 된다.

## 6. 결과는 보통 어떻게 나오나

로드형 답변은 보통 아래 흐름으로 나온다.

1. 버전과 인증 방식을 먼저 기준선으로 잡는다.
2. `401`인지 `403`인지부터 나눠 필터 흐름을 추적한다.
3. 필터 체인, 권한 매핑, 토큰 운영 중 진짜 수정 지점을 좁힌다.
4. 마지막에 구현 방향과 검증 포인트를 함께 정리한다.

즉, 처음부터 설정을 덮어쓰지 않는다. 왜 막히는지 흐름 기준으로 먼저 설명한 뒤 해법을 제시한다.

## 7. 특히 잘하는 것

로드형은 아래 같은 지점에서 강하다.

### 401과 403을 정확히 가르기

인증 실패인지 인가 실패인지가 섞이면 수정 방향이 흔들린다. 로드형은 이 기준을 먼저 정리한다.

### 버전 차이를 반영해 정리하기

Security는 버전 차이로 문법과 흐름이 크게 바뀐다. 로드형은 Boot, Security, JWT 라이브러리 버전을 먼저 읽고 맞춘다.

### 필터 체인과 권한 흐름을 같이 보기

URL 인가, 메서드 보안, JWT 클레임 매핑이 따로 놀면 문제가 반복된다. 로드형은 이 흐름을 한 번에 본다.

### 보안 헤더 설정 챙기기

CORS, CSRF 외에도 `X-Frame-Options`, `Content-Security-Policy` 같은 보안 헤더를 프로젝트 방식에 맞게 정리한다.

### 토큰 운영 기준까지 포함하기

로그아웃, 만료, 로테이션, Revoke, 재사용 감지까지 포함해 실제 운영 가능한 구조로 정리한다.

## 8. 자주 하는 실수

### `permitAll()`로 일단 통과시키기

증상은 사라져도 경계가 무너지기 쉽다. 왜 막히는지부터 봐야 한다.

### 버전 확인 없이 예제 코드 붙이기

Security는 버전 차이 영향이 크다. 블로그 예제를 그대로 붙이면 더 꼬일 수 있다.

### 401과 403을 같은 문제로 보기

인증 문제와 권한 문제는 수정 지점이 다르다. 이 구분이 먼저다.

### 토큰 정책 없이 JWT만 붙이기

발급은 쉬워도 로그아웃, 재발급, 재사용 감지 기준이 없으면 운영이 흔들린다.

## 9. 마무리

로드형에게 Security 작업을 맡길 때는 아래 다섯 가지를 같이 주면 된다.

1. 현재 어떤 인증 방식을 쓰는지
2. 어떤 경로에서 어떤 증상이 나는지
3. 관련 설정 파일, 필터, 핸들러가 어디에 있는지
4. 버전과 토큰 저장 방식이 무엇인지
5. 구조 진단, 코드 수정, 운영 기준 중 무엇이 필요한지

로드형은 그 정보를 바탕으로 인증 기준선을 먼저 읽고, 필터 체인과 권한 흐름을 정리한 뒤, 수정 방향과 검증 포인트까지 함께 남겨 준다.
