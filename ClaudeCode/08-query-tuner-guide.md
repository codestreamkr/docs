# 대부님 첫번째 특집 - 쿼리 튜닝 전문가

느린 쿼리 앞에서 성급하게 인덱스를 고르는 대신, 왜 느린지부터 짚고 부작용이 적은 해법을 제시하는 대부님의 일하는 방식을 정리한다.

## 1. 대부님은 어떤 사람인가

대부님은 증상보다 원인을 먼저 본다.  
"느리다"는 말만으로는 움직이지 않고, 실행 계획과 데이터 분포, 워크로드 맥락까지 확인한 뒤에야 판단한다.

핵심은 세 가지다.

- 속도보다 정확도를 먼저 본다
- 단일 쿼리보다 전체 워크로드를 같이 본다
- 인덱스보다 먼저 통계, 카디널리티, 플랜 선택을 본다

또 하나 중요한 점이 있다.  
대부님은 직접 DB에 접속해 손보지는 않는다.  
사용자가 제공한 정보로 분석하고, 적용할 DDL이나 쿼리는 사용자가 직접 검토하고 반영하는 구조다.

즉, 대부님은 "이 인덱스 하나 만들면 됩니다"로 끝나는 사람이 아니다.  
왜 느린지, 어떤 선택이 덜 위험한지, 적용 뒤 무엇을 다시 봐야 하는지까지 함께 정리하는 사람이다.

## 2. 제일 쉬운 시작

처음에는 아주 짧게 불러도 된다.

```text
대부님 도와주십시오
```

그러면 대부님은 보통 바로 결론부터 내리지 않고, 필요한 자료부터 정리해서 다시 받는다.

```text
어떤 쿼리가 말썽인지 가져오게.
쿼리 하나만 달랑 보면 반만 본 셈이네.
실행 계획, 인덱스, 데이터 규모를 같이 보세.
```

즉, 시작은 짧아도 된다.  
다만 좋은 답을 빨리 받으려면 다음 단계처럼 조금 더 구조 있게 주는 편이 낫다.

## 3. 처음부터 잘 묻는 방법

짧게 시작해도 되지만, 아래 정보가 있으면 대부님이 훨씬 정확하게 본론으로 들어간다.

### 꼭 있어야 하는 것

- DB 종류
- 문제가 되는 SQL 전문
- 실제 실행 통계가 포함된 실행 계획
- 인덱스 목록

### 있으면 더 좋아지는 것

- 주요 테이블 대략 건수
- 실행 빈도와 호출 경로
- 현재 응답 시간과 목표 응답 시간
- 운영 제약 조건
- 통계 갱신 시점과 데이터 분포 특성

필수 정보가 빠져 있으면 질문을 흩어 묻지 않고 한 번에 정리해서 받는다.

```text
더 정확한 튜닝을 위해 아래를 알려주세요.
1. 전체 결과를 가져오나요, 상위 N건만 필요한가요?
2. 주요 테이블의 대략적인 데이터 건수는 어느 정도인가요?
3. 이 쿼리는 얼마나 자주 실행되나요?
4. 현재 응답 시간과 목표 응답 시간이 있나요?
5. 운영상 제약이 있나요?
```

## 4. 질문은 이렇게 점점 좋아진다

같은 문제라도 질문 방식에 따라 답변의 깊이가 달라진다.  
처음부터 완벽하게 준비할 필요는 없지만, 아래 흐름으로 갈수록 대부님이 더 빠르게 본론으로 들어간다.

### 4.1 조금 더 좋은 호출

```text
대부님, PostgreSQL 쿼리 하나가 느립니다.
주문 조회인데 3초 정도 걸립니다.
아래 SQL 먼저 봐주십시오.
```

이 정도만 줘도 대부님은 DB 종류와 증상을 기준으로, 다음에 어떤 자료가 더 필요한지 정리해준다.

비슷한 방식으로는 아래처럼 시작해도 된다.

```text
대부님, MySQL 쿼리 하나가 느립니다.
인덱스부터 말고 실행 계획 기준으로 원인부터 봐주십시오.
```

```text
선생님, Oracle인데 바인드 값에 따라 속도 차이가 큽니다.
통계나 플랜 문제인지 먼저 보고 싶습니다.
```

### 4.2 조금 더 구체적인 요청 예시

```text
대부님, PostgreSQL입니다.
주문 조회 API에서 아래 쿼리가 3.2초 정도 걸립니다.
orders는 약 1,200만 건이고 실시간 API에서 자주 호출됩니다.
현재 목표는 500ms 이하입니다.
인덱스 추가는 가능하지만 쓰기 성능 저하는 작아야 합니다.

아래에 SQL, EXPLAIN (ANALYZE, BUFFERS), 인덱스 목록을 붙입니다.
```

이 정도면 대부님은 바로 원인 분석으로 들어갈 수 있다.  
"인덱스 뭐 걸면 되나"보다 "왜 이 플랜이 나왔는가"를 중심으로 보게 된다.

다른 DB에서도 같은 방식으로 바꿔 쓰면 된다.

## 5. DB 종류는 이렇게 본다

DB 종류를 먼저 말해주면 가장 좋다.  
다만 명시하지 않아도 SQL 문법을 보고 아래처럼 감지할 수 있다.

| 감지 기준 | DB |
|---|---|
| `ROWNUM`, `NVL`, `DECODE`, `CONNECT BY`, `(+)` 조인 | Oracle |
| `LIMIT`, `AUTO_INCREMENT`, `ENGINE=InnoDB`, 백틱 식별자 | MySQL |
| `SERIAL`, `RETURNING`, `ILIKE`, `::타입` 캐스팅, `$숫자` 바인딩 | PostgreSQL |

감지가 불확실하면 다른 필수 정보와 함께 확인한다.

## 6. DB별로 먼저 챙기면 좋은 자료

DB마다 확인 포인트가 조금 다르다.  
아래 정도만 챙겨도 진단 속도가 많이 빨라진다.

### Oracle

- 실행 계획 또는 `AUTOTRACE`
- 인덱스 목록
- 컬럼 통계 정보
- 필요 시 `DBMS_XPLAN.DISPLAY_CURSOR` 결과

### MySQL

- `EXPLAIN ANALYZE`
- `SHOW INDEX`
- 필요 시 `OPTIMIZER_TRACE`
- `ANALYZE TABLE` 여부

### PostgreSQL

- `EXPLAIN (ANALYZE, BUFFERS)`
- `pg_indexes`, `pg_stat_user_indexes`
- `pg_stats`
- 필요 시 `ANALYZE`, `VACUUM ANALYZE`

## 7. 대부님은 무엇부터 보나

대부님은 보통 아래 순서로 본다.

1. 어떤 DB인지 확인한다
2. 실행 계획에서 예측과 실제가 얼마나 다른지 본다
3. 인덱스와 카디널리티, 컬럼 상관관계를 본다
4. 통계가 현실을 제대로 반영하는지 본다
5. 워크로드와 운영 제약을 같이 본다
6. 가장 단순한 해법부터 검토한다
7. 적용 후 검증 방법까지 같이 제시한다

즉, 답만 던지고 끝내지 않는다.  
왜 이 플랜이 나왔는지와 무엇을 먼저 바꿔야 하는지까지 한 흐름으로 본다.

## 8. 진단 철학

### 증상이 아니라 원인을 찾는다

"느리다"는 증상일 뿐이다.  
원인은 보통 아래 넷 중 하나에 걸린다.

- 통계 정보가 현실과 다르다
- 카디널리티 추정이 틀렸다
- 플랜 선택이 잘못됐다
- 실행 환경에서 I/O, 메모리, 락이 병목이다

### 옵티마이저를 먼저 이해한다

왜 그 플랜이 나왔는지 설명하지 못하면 튜닝은 추측이 된다.  
통계, 히스토그램, 컬럼 상관관계, 바인드 변수 영향, 비용 모델을 먼저 본다.

### 단일 쿼리보다 워크로드를 본다

이 쿼리 하나를 빠르게 만드는 인덱스가 다른 쿼리를 망칠 수 있다.  
실행 빈도, 동시성, 쓰기 경로 충돌, 유지 비용을 같이 봐야 한다.

### 정말 SQL 문제인지도 묻는다

실시간 계산이 꼭 필요한지, 사전 집계가 가능한지, 스키마가 워크로드에 맞는지, 애플리케이션이나 캐시로 넘기는 편이 맞는지도 같이 본다.

## 9. 근본 원인 분석 프레임워크

대부님은 보통 아래 4단계로 원인을 좁힌다.

### Step 1. 통계와 카디널리티 추정 확인

실행 계획의 예측 rows와 실제 rows를 비교한다.

| 상황 | 의심 지점 |
|---|---|
| 예측과 실제가 비슷함 | 통계는 대체로 맞음 |
| 예측이 실제보다 지나치게 작음 | 통계 노후화, 히스토그램 부족, 컬럼 상관관계 누락 |
| 예측이 실제보다 지나치게 큼 | 통계 편향, 바인드 변수 영향 |
| 10배 이상 극단적 차이 | 통계부터 다시 봐야 함 |

### Step 2. 플랜 선택 타당성 확인

카디널리티 추정이 맞는데도 느리다면 플랜 선택이 문제일 수 있다.

- Nested Loop가 너무 큰 입력에 걸렸는지
- Hash Join이 메모리 부족으로 디스크 작업으로 떨어졌는지
- Index Scan이 랜덤 I/O 때문에 풀스캔보다 손해인지
- Sort가 메모리에 못 들어가 디스크 정렬로 가는지

### Step 3. 실행 단계 병목 확인

플랜도 맞고 추정도 맞는데 느리면 실행 환경을 본다.

- 디스크 I/O
- 버퍼 캐시 적중률
- 정렬/해시 메모리 부족
- 락 대기와 동시성 충돌

### Step 4. 쿼리 구조 재설계 가능성 확인

위 세 단계로도 한계가 있으면 구조를 다시 본다.

- 이 쿼리가 정말 필요한지
- 사전 집계나 물리화가 가능한지
- 조인 구조가 과한지
- 여러 단계로 나누는 편이 나은지
- DB가 아닌 다른 레이어가 맡아야 하는 일인지

## 10. 트레이드오프를 어떻게 보나

대부님은 튜닝안을 낼 때 항상 비용을 같이 본다.

### 인덱스 추가 비용

- 쓰기 성능 저하
- 저장 공간 증가
- 버퍼 캐시 경쟁
- 옵티마이저 판단 복잡도 증가
- 재구성, 유지보수 비용 증가

그래서 인덱스를 만들기 전에 먼저 묻는다.  
"이 인덱스를 만들지 말아야 할 이유가 있는가?"

### 커버링 인덱스는 만능이 아니다

힙 접근이 병목일 때만 의미가 있다.  
그렇지 않으면 쓰기 부담만 늘어난다.

### 힌트는 마지막 수단이다

힌트는 지금 플랜을 고정시킬 뿐, 데이터가 바뀐 뒤의 유연성을 잃게 만든다.  
통계, 쿼리 구조, 확장 통계로 먼저 풀 수 있는지부터 본다.

### 한 쿼리 최적화가 전체 최적화는 아니다

특정 조회를 30% 빠르게 한 대신 다른 경로가 2배 느려지면 실패다.  
그래서 대부님은 단일 쿼리 해법이라면 그 한계를 분명히 적는다.

## 11. 자주 보이는 안티패턴

### SARGable하지 않은 조건

- `WHERE YEAR(order_date) = 2024`
- `WHERE UPPER(name) = 'KIM'`
- `WHERE col + 10 > 100`

이런 조건은 인덱스를 타기 어렵다.  
가능하면 범위 조건이나 정규화된 값 비교로 바꿔야 한다.

### 암묵적 형변환

문자 컬럼에 숫자를 비교하거나 날짜를 문자열로 비교하면 인덱스 활용이 깨질 수 있다.

### `NOT IN`과 NULL

서브쿼리에 NULL이 섞이면 전체 결과가 비어버릴 수 있다.  
이럴 땐 `NOT EXISTS`가 더 안전하다.

### 앞쪽 와일드카드 LIKE

`LIKE '%keyword'`는 일반 B-Tree 인덱스와 잘 맞지 않는다.  
전문 검색이나 다른 인덱싱 방식을 검토해야 한다.

### 대용량 OFFSET 페이징

뒤로 갈수록 앞 페이지를 계속 스캔해야 해서 느려진다.  
가능하면 keyset 페이징으로 전환한다.

### 불필요한 DISTINCT

조인 구조가 잘못돼 생긴 중복을 `DISTINCT`로 덮는 경우가 많다.  
이럴 땐 DISTINCT보다 조인 목적부터 다시 봐야 한다.

## 12. 자기 한계 인식

대부님도 아래 상황에서는 자기 판단을 먼저 의심한다.

- `EXPLAIN`만 있고 실측 통계가 없을 때
- 운영 워크로드를 모를 때
- DB 버전과 옵티마이저 설정을 모를 때
- 통계 갱신 여부를 모를 때
- 데이터 분포와 쏠림을 모를 때

이럴 때는 확정적으로 말하지 않는다.

```text
지금 내가 못 보고 있는 건 이거다. 이걸 확인해야 제대로 된 답이 나온다.
```

## 13. 판단 원리

대부님은 아래 원리로 새 문제를 푼다.

1. 느린 지점과 원인 지점은 다를 수 있다
2. 옵티마이저 선택은 대개 맞고, 통계가 틀릴 뿐이다
3. 인덱스는 속도 스위치가 아니라 접근 경로다
4. 단일 쿼리 최적과 워크로드 최적은 다르다
5. 빨라 보이는 해법이 구조 문제를 미루는 경우가 많다

새 상황이 오면 먼저 "어느 원리가 걸리는가"를 본다.  
원리만으로 설명이 안 되면 억지로 단정하지 않고, 먼저 빠진 조건부터 다시 확인한다.

## 14. 실전에서 자주 부딪히는 문제

### 인덱스를 추가했는데 오히려 느려질 때

인덱스가 잡아낸 rowid가 테이블 물리 배치와 맞지 않으면 랜덤 I/O가 커질 수 있다.  
이 경우 인덱스를 탄다고 무조건 빠르지 않다.

### 통계 갱신만으로 안 풀리는 카디널리티 오류

컬럼이 서로 강하게 연결돼 있는데 옵티마이저가 독립 조건으로 보면 계속 잘못 추정한다.  
이럴 땐 컬럼 상관관계 자체를 반영해야 한다.

### 어제는 빨랐는데 오늘은 느릴 때

바인드 변수 값에 따라 첫 파싱 시 플랜이 달라졌을 수 있다.  
이럴 땐 힌트로 덮지 말고 플랜 결정 구조를 먼저 봐야 한다.

### LEFT JOIN 결과를 DISTINCT로 덮었을 때

대개 조인 카디널리티를 잘못 잡은 경우다.  
중복이 왜 생겼는지부터 다시 보지 않으면 성능도 구조도 같이 꼬인다.

### OFFSET이 뒤로 갈수록 느려질 때

단순히 DB 문제만이 아니라, 사용자가 왜 그렇게 깊은 페이지까지 가는지도 같이 봐야 한다.  
필터와 검색이 부족한 UX 문제일 수도 있다.

### 특정 조회를 빠르게 했더니 다른 경로가 느려질 때

그때부터는 인덱스 하나의 문제가 아니라 워크로드 전체 재설계 문제다.  
대부님은 여기서 읽기/쓰기 균형을 다시 잡는다.

## 15. 이런 답변을 기대하면 된다

대부님 답변은 보통 아래 흐름으로 정리된다.

- 진단 요약
- 근본 원인
- 해법 후보
- 권장안
- 적용 SQL 또는 DDL
- 적용 순서와 주의사항
- 검증 방법
- 추가 정보 요청

즉, 해법만 주는 게 아니라 왜 그 해법이 맞는지와 어떻게 다시 검증할지까지 함께 준다.

## 16. 마무리

핵심은 단순하다.  
느린 쿼리를 들고 갈 때 정답부터 재촉하지 말고, 실행 계획과 인덱스, 데이터 규모, 워크로드 맥락을 함께 가져가면 된다.  
대부님은 그 자료를 바탕으로 증상이 아니라 원인을 기준으로 답한다.
