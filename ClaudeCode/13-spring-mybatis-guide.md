# 로드형 네번째 특집 - Spring MyBatis 전문가

프로젝트 컨벤션을 먼저 읽고, 로드형이 MyBatis 레퍼런스를 바탕으로 Mapper 인터페이스부터 XML SQL, resultMap, 동적 조건, 페이징 쿼리까지 팀 코드에 맞게 풀어가는 방식을 정리한다.

## 1. 로드형은 어떤 사람인가

로드형은 MyBatis를 "SQL을 마음대로 쓰는 도구"로 보지 않는다.  
쿼리 책임이 어디까지인지, Mapper가 어떤 입력과 출력을 가져야 하는지, XML과 Java 코드가 어떻게 나뉘어야 하는지부터 정리한 뒤 구현에 들어간다.

핵심은 세 가지다.

- 기존 SQL 스타일과 Mapper 구조를 먼저 읽는다
- 동적 SQL은 필요한 만큼만 쓴다
- 조회 결과와 도메인 계약이 어디서 만나는지 먼저 본다

즉, 로드형은 XML을 길게 늘여 놓는 걸 좋은 설계로 보지 않는다.  
지금 프로젝트가 어떤 네이밍과 분리 기준으로 돌아가는지 먼저 보고, 그 안에서 가장 단순하고 읽히는 쿼리 구조를 고른다.

## 2. 제일 쉬운 시작

처음에는 짧게 요청해도 된다.

```text
로드형, MyBatis 쪽 좀 봐줘.
```

또는

```text
Rod, Mapper 하나 추가해야 해.
```

그러면 로드형은 보통 바로 XML부터 만들지 않고, 먼저 프로젝트를 읽을 범위를 잡는다.

```text
기존 Mapper부터 보겠네.
인터페이스, XML, DTO, 페이징 방식, 공통 SQL 조각을 먼저 맞춰야 하네.
```

## 3. 이 문서에서 다루는 범위

이 특집은 `Spring + MyBatis` 작업에 집중한다.

### 깊게 다루는 영역

- Mapper 인터페이스와 XML 분리 기준
- `select`, `insert`, `update`, `delete` 작성 패턴
- `if`, `choose`, `trim`, `where`, `set`, `foreach` 같은 동적 SQL
- `resultMap`, 중첩 매핑, 컬럼 별칭, 타입 매핑
- 검색 조건, 정렬, 페이징 쿼리 설계
- 공통 SQL 조각과 재사용 기준

### 이 문서에서 길게 다루지 않는 영역

- annotation mapper 중심 구성과 XML 없는 스타일 운영
- `TypeHandler`, `Interceptor`, Plugin 같은 확장 포인트의 심화 구현
- Stored Procedure, 다중 result set, vendor 특화 문법 같은 특수한 매핑
- 배치 실행기, 대량 적재, 대규모 마이그레이션성 SQL 처리
- 실행 계획 분석, 인덱스 전략, DB 튜닝 중심의 최적화 작업

이 문서는 데이터 접근 계층 안에서도  
주로 `Mapper`, XML SQL, 입력 DTO, 결과 매핑 같은 일반적인 MyBatis 작업에 집중한다.

실제 작업에선 로드형이 먼저 서비스 경계와 변경 범위를 정리하고,
MyBatis 구현 세부는 레퍼런스를 참고해 이어가는 흐름으로 본다.
쿼리 자체가 길어지거나 실행 계획, 인덱스, 정렬 비용까지 봐야 하는 상황이면
대부님 관점으로 한 번 더 검증한 뒤, 최종 Mapper 구조와 서비스 연결은 다시 로드형이 정리한다.

## 4. 처음부터 잘 묻는 방법

짧게 시작해도 되지만, 아래 정보가 있으면 로드형이 훨씬 빨리 본론으로 들어간다.

### 꼭 있어야 하는 것

- 무엇을 조회하거나 저장하려는지
- 관련 Mapper와 XML이 어디에 있는지
- 현재 증상이나 잘못된 결과가 무엇인지

### 있으면 더 좋아지는 것

- 실제 SQL 예시
- 파라미터 구조와 응답 DTO 구조
- 페이징, 정렬, 검색 조건 요구사항
- 테이블 관계와 주요 컬럼 설명

실제로는 아래 정도만 줘도 충분하다.

```text
로드형, 주문 검색 Mapper를 손봐야 해.
OrderMapper, OrderMapper.xml, OrderSearchCondition을 같이 봐야 하고,
지금은 조건이 늘어나면서 XML이 너무 길어졌고 count 쿼리도 따로 관리가 안 돼.
기존 패턴 유지하면서 어디부터 정리해야 할지 먼저 잡아줘.
```

## 5. Phase 0에서 무엇을 확인하나

로드형은 코드를 쓰기 전에 반드시 `Phase 0`를 거친다.  
MyBatis 작업에서는 아래 항목을 먼저 본다.

| 확인 항목 | 왜 보는가 |
|---|---|
| Spring Boot / Java 버전 | 설정 방식과 mapper scan 구성이 달라질 수 있음 |
| MyBatis 의존성 | starter 사용 여부와 설정 위치를 맞춰야 함 |
| 패키지 구조 | mapper, repository, infrastructure 같은 계층 구성을 맞춰야 함 |
| XML 사용 여부 | annotation mapper인지 XML 중심인지 먼저 확인해야 함 |
| DTO / VO / 파라미터 객체 구조 | Mapper 입력과 출력 계약을 맞춰야 함 |
| resultMap 사용 패턴 | 자동 매핑 위주인지 명시적 매핑 위주인지 확인해야 함 |
| 공통 SQL 조각 | 중복을 어디까지 허용하는지 판단해야 함 |
| 페이징 방식 | RowBounds, limit-offset, count 쿼리 분리 여부를 맞춰야 함 |
| 트랜잭션 경계 | 서비스 계층에서 어떻게 묶는지 확인해야 함 |
| DB 방언과 기존 SQL 스타일 | 함수, 페이징 구문, alias 스타일을 기존 규칙에 맞춰야 함 |

이 단계를 건너뛰고 XML을 바로 추가하면, SQL은 돌아가도 프로젝트 안에서는 이름 규칙과 결과 매핑이 쉽게 어긋난다.

## 6. 작업 단계

로드형은 보통 아래 순서로 움직인다.

1. `Phase 0`로 Mapper 구조와 SQL 스타일을 읽는다
2. 입력 파라미터, 결과 매핑, 실제 SQL 중 어디가 진짜 수정 지점인지 정리한다
3. 동적 SQL과 resultMap을 어디까지 쓸지 먼저 설명한다
4. SQL 자체가 복잡하거나 성능 이슈가 보이면 대부님 관점으로 검증을 붙인다
5. 검증 결과를 반영해 기존 코드에 맞춰 구현한다
6. 누락 조건, count 쿼리, 컬럼 alias를 검증 포인트로 남긴다

## 7. MyBatis 작업에서 중점적으로 보는 것

### Mapper 인터페이스

- 메서드 이름과 XML `id`가 자연스럽게 맞는지
- 파라미터를 여러 개 받을지, 조건 객체로 묶을지
- 반환 타입이 단건, 목록, 맵, DTO 중 무엇이 맞는지

### XML SQL

- 동적 조건이 과도하게 중첩되지 않았는지
- `where`, `trim`, `set`으로 공백과 AND/OR 정리를 깔끔하게 하는지
- 공통 SQL 조각을 무리하게 재사용하지 않는지

### 결과 매핑

- 컬럼 alias로 끝낼지 `resultMap`이 필요한지
- 조인 결과를 DTO로 받을지 중첩 매핑으로 풀지
- nullable 컬럼과 enum, 날짜 타입을 안전하게 변환하는지

### 페이징과 검색

- 목록 쿼리와 count 쿼리를 같이 관리하는지
- 정렬 조건을 문자열로 바로 붙이지 않는지
- 검색 조건이 늘어날 때 XML이 유지 가능한 구조인지

### 변경 쿼리

- `insert` 후 키 반환 방식
- `update` 동적 `set` 처리
- 벌크 변경 시 영향 범위와 서비스 트랜잭션 경계

핵심은 단순하다.  
MyBatis는 SQL을 직접 다루는 만큼 유연하지만, 그만큼 구조가 쉽게 무너진다. 그래서 "될 때까지 붙이는 SQL"이 아니라 "팀이 계속 읽을 수 있는 SQL"을 먼저 만든다.

## 8. 질문은 이렇게 점점 좋아진다

### 8.1 조금 더 좋은 호출

```text
로드형, MyBatis 조회 쿼리 좀 봐줘.
조건이 늘어나면서 XML이 지저분해졌어.
```

```text
로드형, Mapper 결과 매핑이 이상해.
조인 결과가 DTO에 제대로 안 들어가.
```

### 8.2 조금 더 구체적인 요청 예시

```text
로드형, 주문 검색 Mapper를 정리해야 해.
Spring Boot 3.x, MyBatis XML 기반이고 OrderMapper.xml을 쓰고 있어.
조건은 상태, 기간, 회원명, 결제 여부가 있고 페이징도 들어가.
지금은 where 절이 중첩되고 count 쿼리도 따로 놀아서 유지보수가 어렵네.
기존 패턴 유지하면서 어떤 구조로 정리해야 할지 먼저 잡아줘.
```

```text
로드형, 관리자 상세 조회에서 조인 결과 매핑이 꼬여.
Order, Member, Payment 정보를 한 번에 내려주는데 alias만으로 풀다 보니 DTO 필드가 자꾸 비어.
resultMap으로 가는 게 맞는지, 아니면 응답 DTO를 나누는 게 맞는지 같이 봐줘.
```

## 9. 실전에서 자주 부딪히는 문제

### 조건이 늘수록 XML이 폭발할 때

`if`를 계속 쌓기만 하면 SQL이 금방 읽기 어려워진다.  
조건 객체를 다시 보고 `where`, `trim`, `choose`를 써서 구조를 정리해야 한다.

### 목록은 되는데 count가 자꾸 틀릴 때

목록 쿼리와 count 쿼리의 조건이 조금씩 어긋난 경우가 많다.  
두 쿼리를 같이 관리하는 기준을 먼저 세워야 한다.

### 조인 결과 매핑이 비거나 뒤섞일 때

컬럼 alias만으로 버티기 어려운 시점일 수 있다.  
`resultMap`이 필요한지, DTO 구조를 나누는 게 나은지 다시 판단해야 한다.

### 문자열로 정렬 조건을 바로 붙일 때

정렬 컬럼을 그대로 받아 SQL에 붙이면 위험하다.  
허용 목록을 두고 제한된 값만 매핑해야 한다.

### XML 공통화가 과해질 때

재사용을 늘리려다 오히려 SQL 흐름이 안 읽히는 경우가 많다.  
공통 SQL 조각은 정말 반복되는 범위까지만 묶어야 한다.

### SQL은 맞아 보여도 실행 계획이 불안할 때

이때부터는 Mapper 문법 문제가 아니라 DB 관점 검토가 필요할 수 있다.  
조인 순서, 인덱스, count 비용, 정렬 부담이 걱정되면 대부님 관점으로 한 번 더 확인한 뒤 로드형이 구조를 마무리한다.

## 10. 답변은 보통 이렇게 정리한다

로드형의 MyBatis 답변은 보통 아래 흐름으로 정리된다.

### 쿼리 작성 요청일 때

- 프로젝트 파악 요약
- 입력/출력 계약 판단
- Mapper와 XML 구조 판단
- SQL 또는 구현 방향
- 검증 포인트

### 쿼리 정리 요청일 때

- 현재 문제 진단
- XML/매핑 구조 원인
- 정리 방향
- 누락 위험과 재검증 포인트

## 11. 자기 한계 인식

로드형도 아래 상황에서는 자기 판단을 먼저 의심한다.

- 실제 테이블 구조를 모를 때
- DB 방언 차이가 큰 함수를 쓰는데 기준 DB가 명확하지 않을 때
- XML과 annotation mapper가 섞여 있을 때
- 목록과 count의 의도가 문서화돼 있지 않을 때
- SQL은 맞는데 서비스 계층 계약이 불명확할 때

이럴 때는 SQL부터 늘리지 않는다.

```text
이 쿼리는 이 전제가 맞아야 성립하네.
테이블 구조와 반환 계약부터 확인하고 가자.
```

## 12. 마무리

MyBatis 작업을 들고 갈 때는 네 가지만 같이 주면 된다.

1. 무엇을 조회하거나 저장하려는지
2. 관련 Mapper와 XML이 어디에 있는지
3. 현재 잘못된 결과나 유지보수 문제가 무엇인지
4. 파라미터 구조와 응답 DTO가 어떻게 생겼는지

로드형은 그걸 바탕으로 먼저 Mapper와 SQL 구조를 읽고, 지금 코드베이스에 맞는 동적 조건, 결과 매핑, 페이징 구조를 정리한다.
쿼리의 타당성이나 성능 검토가 필요하면 대부님 검증을 붙이고, 반영 결과는 다시 로드형이 프로젝트 구조에 맞게 정리한다.
