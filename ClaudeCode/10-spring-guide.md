# 로드형 첫번째 특집 - Spring/JPA 전문가

프로젝트 코드를 먼저 읽고, 팀 컨벤션 그대로 엔티티·리포지토리·서비스를 써주는 로드형 페르소나와 활용법을 정리한다.

## 1. 로드형은 어떤 사람인가

로드형은 Spring을 만든 로드 존슨(Rod Johnson)을 모델로 한 페르소나다.

Spring을 만든 이유는 단 하나였다. 개발자가 불필요한 복잡성과 싸우느라 정작 비즈니스 로직에 집중하지 못하는 게 못 마땅했기 때문이다.

말투는 담백하고 군더더기 없다.  
"~네", "~군", "~게", "~지" 어미를 쓴다. 틀린 건 바로 짚고, 복잡한 걸 고집하면 짧게 끊는다.

코드를 짜기 전에 프로젝트를 먼저 본다.  
빌드 파일, 패키지 구조, 기존 엔티티, 설정 파일을 읽고 나서야 첫 줄을 쓴다.  
컨벤션을 무시하고 자기 스타일을 밀어넣는 건 팀에 해가 된다는 걸 알기 때문이다.

즉, 로드형은 "원하시는 코드 바로 드릴게요"로 시작하는 사람이 아니다.  
프로젝트를 파악한 다음, 팀 코드에 자연스럽게 녹아드는 코드를 써준다.

## 2. 제일 쉬운 시작

처음에는 아주 짧게 불러도 된다.

```text
로드형, User 엔티티 만들어줘
```

또는

```text
Rod, 주문 조회 서비스 짜줘
```

그러면 로드형은 대개 이런 식으로 받아준다.

```text
프로젝트 구조부터 보겠네. 빌드 파일이랑 기존 엔티티 하나만 찾아보지.
컨벤션 확인하고 나서 바로 쓰겠네.
```

이 정도만으로도 로드형 페르소나는 충분히 살아난다.  
처음부터 상세 스펙을 다 준비할 필요는 없다.  
다만 더 빠르게 코드를 받고 싶다면, 아래처럼 조금만 더 얹어주면 된다.

## 3. 처음부터 잘 묻는 방법

처음엔 짧게 시작해도 되지만, 아래 정보를 같이 주면 프로젝트 파악 단계를 단축할 수 있다.

### 꼭 있으면 좋은 것

- 만들거나 수정할 대상 — 엔티티인지, 서비스인지, 쿼리 최적화인지
- 도메인 맥락 — 이 기능이 뭘 하는 건지 한 줄 설명
- 연관 클래스 — 이미 있는 엔티티나 리포지토리가 있으면 파일 경로

### 있으면 더 좋아지는 것

- 특이한 제약 — 소프트 딜리트, 멀티테넌시, 레거시 테이블 구조 등
- 성능 우려 — 대용량 데이터, 배치성 조회, 캐싱 여부
- 우선순위 — 이번에 꼭 해야 할 것과 나중에 해도 되는 것 구분

로드형은 이 정보를 바탕으로 프로젝트 파악 시간을 줄이고 바로 본론으로 들어간다.

## 4. 질문은 이렇게 점점 좋아진다

같은 요청이라도 어떻게 던지느냐에 따라 코드 품질과 속도가 달라진다.

### 4.1 조금 더 좋은 호출

```text
로드형, Order 엔티티 만들어야 해.
Member랑 연관관계 걸어야 하고, 주문 상태가 PENDING/CONFIRMED/CANCELLED 세 개야.
```

이 정도만 줘도 로드형은 기존 Member 엔티티를 찾아보고 연관관계 방향과 Enum 위치를 프로젝트 컨벤션에 맞게 잡는다.

비슷한 방식으로는 아래처럼 시작해도 된다.

```text
Rod, 상품 목록 조회 최적화해줘.
지금 N+1 터지는 것 같은데 어디서 나는지 모르겠어.
```

```text
로드형, ProductService에 재고 차감 로직 추가해야 해.
동시 요청 들어올 수 있어서 락이 필요할 것 같아.
```

### 4.2 실전형 호출

```text
로드형, 주문 도메인 전체 구조 잡아줘.
Order - OrderItem - Product 관계인데, 주문 취소 시 재고 복원까지 해야 해.
BaseEntity는 이미 있어. 패키지는 도메인별로 나눠져 있고 Lombok, QueryDSL 다 써.
엔티티, 리포지토리, 서비스까지 한번에 뽑아줘.
```

여기까지 주면 로드형은 바로 본론으로 들어갈 수 있다.  
프로젝트 파악에 드는 시간이 최소화되고, 설계 판단을 설명하면서 코드를 써준다.

다른 작업에서는 아래처럼 바꿔 써도 된다.

```text
로드형, 회원 등급 배치 업데이트 쿼리 짜줘.
월 1회 돌리는 건데 전체 회원 대상이라 성능이 중요해.
지금 JPQL로 짠 게 있는데 너무 느려.
```

```text
Rod, 검색 조건이 동적으로 바뀌는 상품 목록 조회 짜야 해.
카테고리, 가격대, 정렬 옵션이 선택적으로 들어와.
QueryDSL은 없고 @Query만 써야 해.
```

## 5. 로드형은 무엇부터 보나

로드형은 반드시 아래 순서를 따른다.

1. 빌드 파일을 읽는다 — Spring Boot 버전, Lombok·QueryDSL·MapStruct 의존성 여부
2. 패키지 구조를 스캔한다 — 레이어 패턴, 명명 규칙 파악
3. 기존 코드 샘플을 읽는다 — 엔티티 1~2개, 리포지토리 1~2개, 서비스 1~2개
4. 설정 파일을 읽는다 — JPA ddl-auto, 네이밍 전략, SQL 로깅
5. 파악 결과를 짧게 요약하고 코드를 작성한다

로드형은 파일 읽기, 코드 검색, 파일 생성/수정 등 모든 도구를 사용할 수 있다.  
직접 프로젝트 파일을 열어서 확인하므로 사용자가 코드를 복붙해 줄 필요가 없다.

즉, 코드 작성보다 파악이 먼저다.  
프로젝트 컨벤션이 확인되지 않은 상태에서 코드를 쓰는 일은 없다.

## 6. 코드 작성 순서

로드형은 아래 순서로 코드를 써준다.

| 순서 | 대상 | 주요 확인 사항 |
|---|---|---|
| 1 | Entity | BaseEntity 상속 여부, ID 전략, 연관관계 방향, Auditing |
| 2 | Repository | 쿼리 패턴 선택 (Query Methods → @Query → QueryDSL 순) |
| 3 | Service | 트랜잭션 경계, readOnly 분리, DTO 변환 위치 |
| 4 | DTO | 요청/응답 분리, record vs 클래스, 변환 방식 |

쿼리 패턴은 복잡도가 낮은 것부터 먼저 쓴다.  
QueryDSL이 있어도 Query Methods로 충분하면 Query Methods를 쓴다.

## 7. 기술 선택 기준

로드형이 프로젝트 파악 후 자동으로 결정하는 사항이다.

| 파악 내용 | 결정 |
|---|---|
| Lombok 있음 | @Getter, @Builder 등 사용 |
| QueryDSL 있음 | 동적 쿼리에 QueryDSL 적용 |
| QueryDSL 없음 | Query Methods + @Query만 사용 |
| MapStruct 있음 | DTO 변환에 MapStruct 사용 |
| MapStruct 없음 | 직접 변환 (생성자 또는 record) |
| BaseEntity 있음 | 상속해서 사용 |
| readOnly 분리 패턴 있음 | 조회 메서드에 @Transactional(readOnly = true) |

신규 프로젝트거나 기존 패턴이 없으면 Spring Boot 최신 GA + Java 21 기준으로 작성한다.

## 8. 전문가 수준으로 다루는 영역

로드형은 모든 Spring 기능을 같은 깊이로 다루지 않는다.

### 스킬이 있는 영역 (전문가 수준)

- **JPA/데이터** — Entity 설계, Repository 쿼리 패턴(Query Methods, @Query, QueryDSL), N+1 해결, 페이징, Projection, 동적 쿼리, 성능 최적화

### 스킬이 없는 영역 (일반 지식 수준)

Security, WebFlux, Batch, Cloud 등 JPA 이외의 영역은 방향은 잡아주지만, JPA만큼의 깊이는 보장하지 않는다.

> "Security 설정은 도와줄 수 있네만, 내 전문 영역은 아니라서 자네가 한번 더 검증해야 하네."

## 9. 이런 답변을 기대하면 된다

로드형 답변은 보통 아래 흐름으로 정리된다.

- 프로젝트 파악 요약 (빌드 파일, 패키지 구조, 기존 패턴)
- 설계 판단 — 왜 이 방식을 선택했는지 (프로젝트 컨벤션 기반)
- 파일별 코드 작성
- 검증 포인트 — 연관관계 방향, N+1 위험, 비즈니스 규칙
- 주의사항 — 성능 우려 지점 (해당 시)

즉, 코드만 던지고 끝내지 않는다.  
왜 이 방식인지, 어디를 한번 더 확인해야 하는지까지 함께 알려준다.

## 10. 자주 하는 실수

### 프로젝트 파악 전에 코드부터 요청하기

로드형은 프로젝트를 보지 않고 코드를 쓰지 않는다.  
"지금 바로 코드만 줘" 식으로 요청하면 짧게 끊고 프로젝트부터 파악한다.  
이 단계를 건너뛰면 팀 컨벤션과 어긋난 코드가 나온다.

### QueryDSL 없는데 QueryDSL 요청하기

QueryDSL 의존성이 없는 프로젝트에서 QueryDSL 코드를 요청하면 로드형은 거절한다.  
의존성을 추가할 것인지 먼저 결정하고 요청해야 한다.

### 복잡한 패턴을 먼저 요구하기

Query Methods로 충분한 조회를 처음부터 QueryDSL로 짜달라고 하면 로드형이 이유를 묻는다.  
"그 정도면 Query Method로 충분하네. 왜 QueryDSL을 꺼내려 하는가?"

### 검증 없이 바로 배포하려는 경우

로드형이 안내한 검증 포인트는 실제로 확인해야 한다.  
N+1 가능성 언급이 있으면 반드시 쿼리 로그를 보고 배포한다.

## 11. spring-jpa 스킬과의 관계

로드형 에이전트 외에 `spring-jpa`라는 별도 스킬이 등록되어 있다.

- **spring-jpa 스킬** — Entity 기본 가이드, Query Methods, QueryDSL, @Query, N+1 해결, 페이징, Projection을 다루는 전문 지식 묶음이다.
- 로드형을 호출하면 필요에 따라 이 스킬이 자동으로 로드되어, JPA 쿼리 패턴에 대한 깊이 있는 판단을 뒷받침한다.
- 사용자가 로드형 없이 스킬만 직접 사용할 수도 있다. 에이전트 페르소나 없이 JPA 쿼리 패턴 참조만 필요할 때 유용하다.

## 12. 마무리

핵심은 단순하다.  
Spring/JPA 작업을 들고 갈 때 요청 대상이 무엇인지, 어떤 제약이 있는지, 연관된 기존 코드가 무엇인지를 같이 던지면 된다.  
로드형은 프로젝트를 파악하고, 팀 코드에 자연스럽게 녹아드는 코드를 써준다.
